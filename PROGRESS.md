# gomoku-online 작업 기록

이 파일은 새 대화에서 이어서 작업할 수 있도록 지금까지 한 작업을 정리한 문서입니다.
최신 커밋: `ac369ef` (master 브랜치, origin과 동기화됨)

## 프로젝트 개요

- 오목 + 렌주룰 + "4턴마다 증강체 선택"을 결합한 로그라이크 스타일 웹 게임
- 스택: Next.js (App Router), Supabase(온라인 대전용 실시간 동기화), Tailwind
- 모드 3가지: `/local`(로컬 패스앤플레이), `/online`(링크 초대), `/online/quick`(빠른 매칭)
- 핵심 파일
  - `lib/gomokuEngine.js` — 순수 보드/렌주룰/증강체 목록(AUGMENTS) 로직, DOM 의존 없음
  - `lib/gameReducer.js` — 모든 게임 상태 전이를 처리하는 단일 리듀서 (로컬/온라인 공용)
  - `components/GomokuBoard.jsx` — 캔버스 렌더링 + 클릭 좌표 변환
  - `components/AugmentPanel.jsx` — 보유 증강체 표시 + 액티브 능력 버튼
  - `components/AugmentSelectOverlay.jsx` — 4턴마다 뜨는 증강 선택 화면 (구 DraftOverlay)
  - `components/RoomClient.jsx` — 온라인 모드 (Supabase 구독)
  - `app/local/page.js` — 로컬 모드 (useReducer)
  - `components/WinOverlay.jsx` — 승리 화면 + 상호 재도전 버튼

## 이번 세션에서 한 일 (시간 순)

### 1. 기존 피드백 18개 반영 (커밋 `1b1ace8`)
사용자가 스트리밍하며 남긴 초기 피드백 18개를 처리:
- 드래프트 → "증강 선택" 용어 통일 (컴포넌트명 `DraftOverlay` → `AugmentSelectOverlay`, CSS 클래스 포함)
- 증강 선택 등급 동기화 도입 (처음엔 3장 혼합 등급 → 나중에 "회차당 단일 등급"으로 재수정, 아래 참고)
- 질풍노도(rush) 너프: 매턴 2개 → 2턴에 1번만 2개
- 대각선강화/일자진: 초반에만 등장하도록 제한
- 온라인 모드에서 상대가 고르는 중인 증강 카드 내용 숨김
- 금지칸 표시 구분(내가 막힌 칸 vs 내가 상대를 막은 칸), 다중 선택 중 진행 상황 표시
- 증강 선택 화면에 "누르고 있으면 판 보기" 눈 아이콘 추가
- 신규 증강: 속박(bind), 오델로(othello), 무위전변(colorSwap), 도박(gamble), 부적(talisman), 액자 완성(→ 나중에 origin의 squareFrame과 통합되며 제거됨)

### 2. origin/master 대규모 병합 (커밋 `21eceab`)
- 같은 베이스 커밋에서 origin이 독립적으로 증강 10개 + 온라인 빠른매칭 + 효과음을 이미 추가한 상태였음
- 충돌 6개 파일(`gomokuEngine.js`, `gameReducer.js`, `RoomClient.jsx`, `GomokuBoard.jsx`, `AugmentPanel.jsx`, `app/local/page.js`)을 수작업으로 병합
- origin의 10개 증강(squareFrame, attrition="물량전", awakening="각성", coinFlip, bind, stinginess, barter, watchtower, ultimatum, leverage)을 그대로 채택
- 중복 개념은 제거: 우리가 만든 3x3 "액자 완성"은 origin의 4x4 "네모"(squareFrame)로 통합, 우리 "속박"/"감시탑" 로직은 origin 버전으로 통일
- **물량전(attrition)이 진짜 "물량전"이었음** — 초기 피드백에서 "물량전 실버 강등"을 요청했을 때 이 증강이 존재하지 않아 doubleMove를 잘못 강등했었는데, 병합 후 attrition에 실버 강등을 재적용하고 doubleMove는 골드로 원복

### 3. 등급 동기화 재설계 (커밋 `25ad1a8`)
- 처음 구현은 "그 회차 3장이 [골드,골드,프리즘] 같은 혼합 조합"을 양쪽이 공유하는 방식이었음
- 사용자 피드백: "그게 아니라 이번 회차는 카드 3장이 통째로 같은 등급(예: 다 골드)이길 원했다"
- `ROUND_TIER_POOL = ["silver","silver","silver","gold","gold","prism"]`에서 회차당 등급 하나를 뽑아 `draftTierPlan[roundIndex]`에 저장, 양쪽 플레이어가 같은 회차엔 항상 같은 단일 등급을 받음

### 4. 신규 증강 30개 브레인스토밍 → 18개로 압축 → 구현 (커밋 `0d442c2`)
사용자가 "신박한 거 30개 가져와봐" 요청 → 30개 제시 → "기존 증강이랑 겹치는 거 빼고 가져와봐" 요청 → 13개 제거하고 17개 확정 → 유저가 "판 뒤엎기" 아이디어 추가 + "붕괴"/"교도소" 스펙 조정 → 최종 18개 구현.

**신규 실버(8)**: 직감(intuition), 균형(balance), 도장깨기(jailbreak), 여진(aftershock), 늦둥이(lateBloomer), 맞불(counterStrike), 축적(stockpile), 잠복(ambush)
**신규 골드(5)**: 습격(raid), 재배치(relocate), 봉인(lockdown), 역병(plague), 성역(sanctuary)
**신규 프리즘(5)**: 판 뒤엎기(boardFlip), 붕괴(collapse), 신탁(oracle), 교도소(prison), 도미노(domino)

구현 메모:
- **교도소(prison)**: `state.prisonActive` 전역 플래그. `getActiveAugmentIds(state, player)`가 켜져 있으면 소유 증강 id 목록에서 프리즘 등급을 전부 걸러냄. `checkImmunity`(철옹성/부적 체크)도 이 필터를 통과한 목록을 씀. **자기 자신 포함**해서 다 꺼짐.
- **붕괴(collapse)**: 새 UI 안 만들고 기존 "칸 1개 클릭" 패턴 재사용 — 클릭한 칸이 3x3의 중심이 됨
- **재배치(relocate)**: 최초로 만든 2단계 pendingTarget (내 돌 클릭 → 인접 빈칸 클릭). `pendingTarget.sourceCell`로 단계 구분
- **역병(plague)**: "승리 판정에서 그 돌만 영구 무효" 대신 "제거 + 그 칸을 양쪽 다 영원히 착수 불가"로 단순화 구현 (모든 승리 조건 함수를 다 고칠 필요 없음). `state.deadCells` 전역 배열
- **여진/맞불/잠복**: 기존 capture/removeStone/undo/banZone 처리 지점에 자동 발동 훅을 꽂음 (새 액션 타입 안 만듦)

### 5. 2차 피드백 5개 (커밋 `f1a3faf`, `ac369ef`)
1. 대각선강화/일자진이 여전히 사기 → `hasThreeOrMoreInARow(board)` 체크 추가, 판 위에 3목 이상 있으면 그 즉시 뽑기 풀에서 제외 (일부러 3목 만들고 뽑는 편법 방지)
2. 다리 놓기(bridge) 골드 → 프리즘 승급
3. 금지구역(banZone) 5턴 → 2턴 너프
4. 상대 마지막 수를 빨간 사각 테두리로 표시 (`GomokuBoard`의 `lastOpponentMoveCell` prop)
5. 재도전을 상호 확인제로 변경 — `REQUEST_REMATCH` 액션 + `rematchRequested: {1,2}` 상태, 둘 다 눌러야 `initialGameState()`로 리셋됨. 기존 `RESTART` 액션은 제거하고 `WinOverlay`를 흑돌/백돌 개별 버튼으로 교체
6. (후속) 영구 봉쇄(permaBlock)가 금지구역이랑 똑같이 골드였던 걸 발견 → 프리즘으로 승급, 교도소 발동 시 이미 걸린 영구봉쇄도 즉시 풀리도록 `isBlocked` 체크에 `!state.prisonActive` 조건 추가

### 6. 밸런스 훑어보기 + 신규 증강 "링 위에서 싸우자" 추가
사용자가 "밸런스 한번 훑어봐주고 증강 좀 더 내자"고 요청. 코드 기준으로 스캔한 결과:
- **버그로 확인돼서 고침**: 붕괴(collapse)가 `checkImmunity`를 아예 안 거쳐서 철옹성/부적을 무시하고 있었음. raid/plague/banZone은 다 면역 체크를 하는데 collapse만 빠져 있었음 → 상대 돌만 면역 적용해서 지켜지도록 수정(자기 자신 돌은 자폭이라 면역과 무관하게 그대로 삭제)
- **아직 미해결로 남겨둔 이슈들** (사용자가 확답 안 함, 손 안 댐):
  1. 역병(gold)이 영구 봉쇄(prism)가 하는 일(빈칸 영구 봉쇄)을 포함하면서 상대 돌 제거까지 더 하는데 등급이 더 낮음
  2. 신탁(prism)이 "다음 회차 프리즘 확정"을 상대에게도 똑같이 적용해서 프리즘치고 자기 이득이 약함
  3. 성역(gold)이 트리거 조건("내 돌로 4방향 완전 포위된 빈칸")이 너무 까다로움

신규 프리즘 증강 **battleRing("링 위에서 싸우자")** 구현:
- 발동 즉시 판이 4수(양쪽 합산)마다 사방으로 한 겹씩 좁혀 들어가 최종 8x8까지 줄어듦, 게임 끝까지 유지, 양쪽 모두에게 적용
- 사용자 확정 사항: 링 바깥으로 밀려난 칸에 이미 있던 돌은 그대로 둠(제거 안 함, 자리만 막힘) — 단, "일단 보류"라 나중에 바뀔 수 있음. 축소 속도/영역점령과의 상호작용은 "노 프리퍼런스"라 내 추천대로 감(4수당 한 겹, 영역점령과 겹쳐도 그냥 시너지로 인정하고 특별 처리 안 함)
- 구현 방식: `BOARD_SIZE`(15) 배열 자체는 안 건드리고, 영구 봉쇄/역병처럼 "링 바깥 = 착수 금지 칸" 패턴으로 처리. `lib/gomokuEngine.js`에 `getRingLevel`/`getRingBounds`/`isOutsideRing` 순수 함수 추가 (`ringStartMove` 기준 경과 수로 레벨 계산, 레벨당 좌/상은 `ceil(level/2)`, 우/하는 `floor(level/2)`씩 인셋 — 15→8은 홀수 차이라 비대칭 인셋으로 맞춤)
- `gameReducer.js`: `ringActive`/`ringStartMove` 상태 추가, `PICK_AUGMENT`에서 prison과 같은 패턴으로 즉시 발동(`pendingTarget` 없음), `CLICK_CELL`의 `isBlocked` 판정에 `isOutsideRing` 추가
- `GomokuBoard.jsx`: `ringBounds` prop 받아서 캔버스에 바깥 영역 반투명 검은 오버레이 + 주황 테두리로 표시. `app/local/page.js`/`RoomClient.jsx`에서 `getRingBounds(ringStartMove, totalStonesPlaced)`로 계산해서 넘겨줌

### 7. 신규 프리즘 2개 추가: 폭주(chaos) / 백지화(wipeout)
사용자가 "1. 증강 선택한 순간부터 완전 무작위로 돌을 둠 2. 판 뒤엎기처럼 모든 돌 삭제(증강/턴 수는 유지) 3. 철옹성이 무르기도 막아주나?" 3가지를 물어봄.

- **3번(철옹성-무르기)**: 확인해보니 이미 되고 있었음. `undo`(되돌리기)는 원래부터 `checkImmunity`를 거쳐서 철옹성/부적에 막힘([gameReducer.js:877-890](lib/gameReducer.js:877)). 지난 세션에 고친 건 collapse(붕괴)뿐이었고 undo는 처음부터 정상이었음.
- **폭주(chaos, prism)**: 발동 즉시 게임 끝까지 양쪽 모두 조작권 상실. 사용자 확정: 게임 끝날 때까지 계속 유지 / 클릭은 필요하지만 클릭 위치 무시하고 무작위 빈칸에 놓임 / 증강 액티브 능력 버튼도 전부 비활성화.
  - `gameReducer.js`: `chaosActive` 상태 추가, `PICK_AUGMENT`에서 prison/battleRing과 같은 패턴으로 즉시 발동. `CLICK_CELL` 진입부에서 `chaosActive`면 클릭한 (x,y)를 버리고 `pickRandomLegalCell(state, currentPlayer)`가 고른 무작위 합법 칸(금지구역/영구봉쇄/역병칸/링 바깥/렌주룰 금수 전부 제외)으로 치환 - 이후 로직은 그대로 재사용됨
  - UI: `canAct`에 `!chaosActive` 조건 추가해서 능력 버튼 비활성화 (AugmentPanel에 별도 손 안 댐, canAct만으로 충분)
  - **라이브 검증 완료**: 임시로 회차 등급 뽑기를 프리즘 고정으로 바꿔서 실제 브라우저에서 폭주를 뽑고, 이미 돌이 있는 칸을 클릭해도 다른 무작위 빈칸에 놓이며 정상적으로 턴/드래프트가 진행되는 것까지 확인 후 원복함
- **백지화(wipeout, prism)**: 1회 사용, 판 위 돌을 양쪽 다 전부 삭제하지만 보유 증강체와 stonesPlaced(진행도)는 그대로 유지. 사용자 확정: 기존 판 뒤엎기(boardFlip, 자기 돌만 재배치)와는 별도의 새 카드로 추가.
  - `gameReducer.js`의 `USE_ABILITY` 케이스에 boardFlip과 같은 패턴으로 추가 (`board: createEmptyBoard()`, `lastMove` 양쪽 다 null, 턴 넘어감)
  - `ONE_TIME_ABILITY_IDS`와 `AugmentPanel.jsx`의 `ACTIVE_ABILITIES`에 등록해서 사용 버튼이 뜨도록 함
  - 코드 리뷰로 검증(boardFlip/colorSwap과 구조 100% 동일), 프리즘 카드 등장 확률이 낮아 라이브 클릭 테스트까지는 못 함

### 8. 치명적 버그 2개 수정: 질풍노도 드래프트 인플레 / 거래·도박·동전던지기·잠복으로 얻은 즉시발동 증강 무효
사용자가 실제 플레이 중 발견한 버그 2개 제보:

**버그 A - 질풍노도(rush)가 증강 선택을 더 자주 봄**: `newStonesPlaced[currentPlayer] % 4 === 0`으로 증강 선택을 트리거하는데, 질풍노도의 보너스 돌(부스트 턴의 2번째 착수)도 이 카운트에 그대로 포함되고 있었음. 질풍노도는 2턴에 1번꼴로 실제 턴 수보다 더 빨리 4의 배수를 채우게 되어, 소유자가 상대보다 훨씬 자주 증강을 뽑는 스노우볼 버그였음.
- 수정: `isRushBonusStone`(=보너스 돌인지) 플래그를 만들어서, 보너스 돌일 때는 `stonesPlaced` 증가 자체를 건너뜀 (기존 `doubleMoveActive`가 이미 이 패턴을 쓰고 있었음 - 그걸 참고함). 겸사겸사 같은 원인으로 보너스 돌에 금지구역/감시탑 카운트다운까지 이중으로 깎이던 것도 같이 고침.
- **라이브 검증**: 임시로 프리즘 고정 뽑기를 켜서 한쪽에 질풍노도를 붙이고, 보너스 돌 포함 정확히 7번째 "카운트되는" 수를 뒀을 때 드래프트가 안 뜨고(수정 전이라면 여기서 떴어야 함), 8번째에서 정상적으로 뜨는 것까지 확인함

**버그 B - 거래/도박/동전 던지기/잠복으로 얻은 교도소·링 위에서 싸우자·폭주가 발동 안 함**: 교도소/링 위에서 싸우자/폭주는 "뽑는 순간 즉시 발동"하는 효과라 `PICK_AUGMENT` 케이스 안에서만 `prisonActive`/`ringActive`/`chaosActive` 플래그를 세팅하고 있었음. 그런데 거래(barter)·도박(gamble의 gamblePrism1)·동전 던지기(coinFlip)·잠복(ambush)은 전부 `ownedAugments`에 직접 카드를 꽂아 넣는 별도 경로라 이 세 카드 중 하나를 "우연히 얻어도" 카드만 보유할 뿐 실제 효과는 하나도 안 켜지는 조용한 버그였음. 사용자가 정확히 이 케이스(거래로 얻은 교도소)를 신고함.
- 수정: `activateInstantAugments(state, grantedAugments)` 공통 헬퍼를 만들어서 PICK_AUGMENT 본 흐름 + 도박 + 거래 + 동전 던지기 + 잠복(`triggerAmbushIfNeeded`) 전부 이 함수를 거치도록 통일함
- **라이브 검증**: 프리즘 풀을 교도소만 나오게 임시로 고정하고, 실제로 "거래" 카드를 뽑아서 사용 → 획득한 카드가 교도소로 표시되고, 상단에 "🔒 '교도소' 발동 중" 배너까지 정상적으로 뜨는 것을 확인함 (수정 전이었다면 카드만 보유하고 배너는 안 떴을 것)

두 수정 다 테스트용으로 잠깐 바꿔둔 회차 등급 고정/프리즘 풀 고정 코드는 검증 후 전부 원복함.

### 9. UI 개선 3개 + 먼저 보기(peek) 회차 등급 무시 버그 수정
- **총 수 표시 추가**: 턴 표시 아래에 "총 N수" 표시 (로컬/온라인 모드 둘 다). 각성(20수)처럼 총 착수 수 기반 증강을 가늠할 수 있게.
- **"먼저 보기" 설명 명확화**: "다음 선택지 카드 1장을 지금 미리 확정해서 봄" → "다음 증강 선택 때 나올 카드 1장을 지금 미리 확정하고 확인함 (그 카드가 다음 선택지 3장 중 하나로 반드시 포함됨)". 단순 미리보기가 아니라 그 카드가 다음 선택지에 확정으로 끼워진다는 걸 명시.
- **"증강체" → "증강" 전체 치환**: 게임 로직/UI/안내 메시지/홈페이지 문구 전부. 단순 문자열 치환이 아니라 조사도 맞춤("증강체는"→"증강은", "증강체를"→"증강을", "증강체와"→"증강과" 등 - "증강"은 받침이 있어서 "증강체"와 조사가 다름). `PROGRESS.md`는 과거 세션 기록이라 그대로 둠.
- **버그: 먼저 보기(peek)가 회차 등급 동기화를 무시함** - 사용자가 "먼저 보기로 프리즘/골드를 발견하면 상대는 실버 고르고 있을 때 나만 프리즘 선택지가 생기는데 의도한 거냐"고 질문, 확인해보니 진짜 버그였음. `peek`은 고르는 즉시 `drawFromPool`로 카드 1장을 확정 예약(`peekedCard`)하는데, 이 `drawFromPool`이 등급 제한 없이 전체 풀에서 뽑아서, 다음 회차가 실버로 정해져도 예약된 카드가 프리즘/골드일 수 있었음 - 세션 3에서 공들여 만든 "회차 등급 동기화"(같은 회차엔 두 플레이어 다 같은 등급 3장)를 실버 카드 하나로 무력화하는 셈이라 명백한 밸런스 버그.
  - 수정: `resolveRoundTier(state, roundIndex)` 헬퍼로 "회차 등급 확정" 로직을 `buildAugmentChoices`에서 분리하고, peek을 고를 때도 이 함수로 **자신의 다음 회차** 등급을 그 자리에서 미리 확정한 뒤(`draftTierPlan`에 고정), 그 등급 안에서만(`drawOneOfTier`) 예약 카드를 뽑도록 변경. `nextRoundIndex = Math.floor(state.stonesPlaced[player] / 4)` 공식은 기존 신탁(oracle)이 쓰던 것과 동일하게 맞춤.
  - **라이브 검증**: 회차 등급을 실버로 강제 고정한 상태에서 "먼저 보기"를 뽑고 다음 회차(8수)까지 진행 → 예약된 카드를 포함한 3장이 전부 실버로 나오는 것을 확인함 (수정 전이었다면 예약 카드가 골드/프리즘일 수 있었음)

모든 수정 사항은 프리뷰 서버에서 라이브로 검증 후 테스트용 임시 코드를 원복함.

## 알아두면 좋은 설계 패턴

- **회차 등급 공유**: `state.draftTierPlan[roundIndex]`에 그 회차 등급을 캐싱, 양쪽이 같은 roundIndex에 도달하면 같은 값을 재사용. `roundIndex = stonesPlaced[player]/4 - 1`
- **1회용 능력 공통 처리**: `oneTimeUsed[player][abilityId]` 플래그 + `ONE_TIME_ABILITY_IDS` 배열(거래가 한꺼번에 소모시킬 대상 목록)
- **면역 체크**: `checkImmunity(state, targetPlayer)` — 철옹성(무한)과 부적(1회성)을 하나로 통합 판정. 새 "공격형" 증강 만들 때 재사용
- **pendingTarget 패턴**: 칸 선택이 필요한 능력은 전부 `{player, kind, need, selected}` 구조. 2단계가 필요하면(재배치) `sourceCell` 같은 필드를 추가해서 같은 리듀서 케이스 안에서 분기
- **prison 필터링**: `getActiveAugmentIds(state, player)`를 거쳐야만 프리즘 무효화가 반영됨. 새 프리즘 증강을 만들 때 `ownedAugments[player].map(a=>a.id)` 대신 이걸 써야 교도소가 먹힘

## 알려진 제약/후속 검토 거리

- `relocate`/`collapse`/`prison` 등 구조적으로 새로운 메커니즘은 실제 대국에서 뽑힐 때까지 라이브 테스트가 제한적이었음 (역병만 end-to-end로 직접 검증함)
- `역병`은 원안(승리 판정에서 그 돌만 영구 제외)이 아니라 "제거 + 칸 영구 봉쇄"로 단순화 구현됨 — 필요하면 원안대로 재구현 가능하지만 모든 win-check 함수를 고쳐야 함
- 온라인 모드는 각 클라이언트가 로컬로 리듀서를 돌리고 결과만 Supabase에 푸시하는 구조라, 이론상 네트워크 지연 시 레이스가 있을 수 있음 (턴제라 실제로는 거의 안 부딪힘, 지금까지 실제 버그 리포트는 없음)
- `.claude/launch.json`에 `npm run dev` 기준 미리보기 서버 설정이 있음 (포트 3000)
