# gomoku-online 작업 기록

이 파일은 새 대화에서 이어서 작업할 수 있도록 지금까지 한 작업을 정리한 문서입니다.
최신 커밋: `82cd508` (master 브랜치) + 이번 세션 변경사항(아직 커밋 안 함, 아래 "19번"~"23번" 섹션 참고)

## 프로젝트 개요

- 오목 + 렌주룰 + "4턴마다 증강체 선택"을 결합한 로그라이크 스타일 웹 게임
- 스택: Next.js (App Router), Supabase(온라인 대전용 실시간 동기화), Tailwind
- 모드 3가지: `/local`(로컬 패스앤플레이), `/online`(링크 초대), `/online/quick`(빠른 매칭)
- 핵심 파일
  - `lib/gomokuEngine.js` — 순수 보드/렌주룰/증강체 목록(AUGMENTS) 로직, DOM 의존 없음
  - `lib/gameReducer.js` — 모든 게임 상태 전이를 처리하는 단일 리듀서 (로컬/온라인 공용)
  - `components/GomokuBoard.jsx` — 캔버스 렌더링 + 클릭 좌표 변환
  - `components/AugmentPanel.jsx` — 보유 증강체 표시 + 액티브 능력 버튼
  - `components/AugmentSelectOverlay.jsx` — 4턴마다 뜨는 증강 선택 화면 (구 DraftOverlay)
  - `components/RoomClient.jsx` — 온라인 모드 (Supabase 구독)
  - `app/local/page.js` — 로컬 모드 (useReducer)
  - `components/WinOverlay.jsx` — 승리 화면 + 상호 재도전 버튼

## 이번 세션에서 한 일 (시간 순)

### 1. 기존 피드백 18개 반영 (커밋 `1b1ace8`)
사용자가 스트리밍하며 남긴 초기 피드백 18개를 처리:
- 드래프트 → "증강 선택" 용어 통일 (컴포넌트명 `DraftOverlay` → `AugmentSelectOverlay`, CSS 클래스 포함)
- 증강 선택 등급 동기화 도입 (처음엔 3장 혼합 등급 → 나중에 "회차당 단일 등급"으로 재수정, 아래 참고)
- 질풍노도(rush) 너프: 매턴 2개 → 2턴에 1번만 2개
- 대각선강화/일자진: 초반에만 등장하도록 제한
- 온라인 모드에서 상대가 고르는 중인 증강 카드 내용 숨김
- 금지칸 표시 구분(내가 막힌 칸 vs 내가 상대를 막은 칸), 다중 선택 중 진행 상황 표시
- 증강 선택 화면에 "누르고 있으면 판 보기" 눈 아이콘 추가
- 신규 증강: 속박(bind), 오델로(othello), 무위전변(colorSwap), 도박(gamble), 부적(talisman), 액자 완성(→ 나중에 origin의 squareFrame과 통합되며 제거됨)

### 2. origin/master 대규모 병합 (커밋 `21eceab`)
- 같은 베이스 커밋에서 origin이 독립적으로 증강 10개 + 온라인 빠른매칭 + 효과음을 이미 추가한 상태였음
- 충돌 6개 파일(`gomokuEngine.js`, `gameReducer.js`, `RoomClient.jsx`, `GomokuBoard.jsx`, `AugmentPanel.jsx`, `app/local/page.js`)을 수작업으로 병합
- origin의 10개 증강(squareFrame, attrition="물량전", awakening="각성", coinFlip, bind, stinginess, barter, watchtower, ultimatum, leverage)을 그대로 채택
- 중복 개념은 제거: 우리가 만든 3x3 "액자 완성"은 origin의 4x4 "네모"(squareFrame)로 통합, 우리 "속박"/"감시탑" 로직은 origin 버전으로 통일
- **물량전(attrition)이 진짜 "물량전"이었음** — 초기 피드백에서 "물량전 실버 강등"을 요청했을 때 이 증강이 존재하지 않아 doubleMove를 잘못 강등했었는데, 병합 후 attrition에 실버 강등을 재적용하고 doubleMove는 골드로 원복

### 3. 등급 동기화 재설계 (커밋 `25ad1a8`)
- 처음 구현은 "그 회차 3장이 [골드,골드,프리즘] 같은 혼합 조합"을 양쪽이 공유하는 방식이었음
- 사용자 피드백: "그게 아니라 이번 회차는 카드 3장이 통째로 같은 등급(예: 다 골드)이길 원했다"
- `ROUND_TIER_POOL = ["silver","silver","silver","gold","gold","prism"]`에서 회차당 등급 하나를 뽑아 `draftTierPlan[roundIndex]`에 저장, 양쪽 플레이어가 같은 회차엔 항상 같은 단일 등급을 받음

### 4. 신규 증강 30개 브레인스토밍 → 18개로 압축 → 구현 (커밋 `0d442c2`)
사용자가 "신박한 거 30개 가져와봐" 요청 → 30개 제시 → "기존 증강이랑 겹치는 거 빼고 가져와봐" 요청 → 13개 제거하고 17개 확정 → 유저가 "판 뒤엎기" 아이디어 추가 + "붕괴"/"교도소" 스펙 조정 → 최종 18개 구현.

**신규 실버(8)**: 직감(intuition), 균형(balance), 도장깨기(jailbreak), 여진(aftershock), 늦둥이(lateBloomer), 맞불(counterStrike), 축적(stockpile), 잠복(ambush)
**신규 골드(5)**: 습격(raid), 재배치(relocate), 봉인(lockdown), 역병(plague), 성역(sanctuary)
**신규 프리즘(5)**: 판 뒤엎기(boardFlip), 붕괴(collapse), 신탁(oracle), 교도소(prison), 도미노(domino)

구현 메모:
- **교도소(prison)**: `state.prisonActive` 전역 플래그. `getActiveAugmentIds(state, player)`가 켜져 있으면 소유 증강 id 목록에서 프리즘 등급을 전부 걸러냄. `checkImmunity`(철옹성/부적 체크)도 이 필터를 통과한 목록을 씀. **자기 자신 포함**해서 다 꺼짐.
- **붕괴(collapse)**: 새 UI 안 만들고 기존 "칸 1개 클릭" 패턴 재사용 — 클릭한 칸이 3x3의 중심이 됨
- **재배치(relocate)**: 최초로 만든 2단계 pendingTarget (내 돌 클릭 → 인접 빈칸 클릭). `pendingTarget.sourceCell`로 단계 구분
- **역병(plague)**: "승리 판정에서 그 돌만 영구 무효" 대신 "제거 + 그 칸을 양쪽 다 영원히 착수 불가"로 단순화 구현 (모든 승리 조건 함수를 다 고칠 필요 없음). `state.deadCells` 전역 배열
- **여진/맞불/잠복**: 기존 capture/removeStone/undo/banZone 처리 지점에 자동 발동 훅을 꽂음 (새 액션 타입 안 만듦)

### 5. 2차 피드백 5개 (커밋 `f1a3faf`, `ac369ef`)
1. 대각선강화/일자진이 여전히 사기 → `hasThreeOrMoreInARow(board)` 체크 추가, 판 위에 3목 이상 있으면 그 즉시 뽑기 풀에서 제외 (일부러 3목 만들고 뽑는 편법 방지)
2. 다리 놓기(bridge) 골드 → 프리즘 승급
3. 금지구역(banZone) 5턴 → 2턴 너프
4. 상대 마지막 수를 빨간 사각 테두리로 표시 (`GomokuBoard`의 `lastOpponentMoveCell` prop)
5. 재도전을 상호 확인제로 변경 — `REQUEST_REMATCH` 액션 + `rematchRequested: {1,2}` 상태, 둘 다 눌러야 `initialGameState()`로 리셋됨. 기존 `RESTART` 액션은 제거하고 `WinOverlay`를 흑돌/백돌 개별 버튼으로 교체
6. (후속) 영구 봉쇄(permaBlock)가 금지구역이랑 똑같이 골드였던 걸 발견 → 프리즘으로 승급, 교도소 발동 시 이미 걸린 영구봉쇄도 즉시 풀리도록 `isBlocked` 체크에 `!state.prisonActive` 조건 추가

### 6. 밸런스 훑어보기 + 신규 증강 "링 위에서 싸우자" 추가
사용자가 "밸런스 한번 훑어봐주고 증강 좀 더 내자"고 요청. 코드 기준으로 스캔한 결과:
- **버그로 확인돼서 고침**: 붕괴(collapse)가 `checkImmunity`를 아예 안 거쳐서 철옹성/부적을 무시하고 있었음. raid/plague/banZone은 다 면역 체크를 하는데 collapse만 빠져 있었음 → 상대 돌만 면역 적용해서 지켜지도록 수정(자기 자신 돌은 자폭이라 면역과 무관하게 그대로 삭제)
- **아직 미해결로 남겨둔 이슈들** (사용자가 확답 안 함, 손 안 댐):
  1. 역병(gold)이 영구 봉쇄(prism)가 하는 일(빈칸 영구 봉쇄)을 포함하면서 상대 돌 제거까지 더 하는데 등급이 더 낮음
  2. 신탁(prism)이 "다음 회차 프리즘 확정"을 상대에게도 똑같이 적용해서 프리즘치고 자기 이득이 약함
  3. 성역(gold)이 트리거 조건("내 돌로 4방향 완전 포위된 빈칸")이 너무 까다로움

신규 프리즘 증강 **battleRing("링 위에서 싸우자")** 구현:
- 발동 즉시 판이 4수(양쪽 합산)마다 사방으로 한 겹씩 좁혀 들어가 최종 8x8까지 줄어듦, 게임 끝까지 유지, 양쪽 모두에게 적용
- 사용자 확정 사항: 링 바깥으로 밀려난 칸에 이미 있던 돌은 그대로 둠(제거 안 함, 자리만 막힘) — 단, "일단 보류"라 나중에 바뀔 수 있음. 축소 속도/영역점령과의 상호작용은 "노 프리퍼런스"라 내 추천대로 감(4수당 한 겹, 영역점령과 겹쳐도 그냥 시너지로 인정하고 특별 처리 안 함)
- 구현 방식: `BOARD_SIZE`(15) 배열 자체는 안 건드리고, 영구 봉쇄/역병처럼 "링 바깥 = 착수 금지 칸" 패턴으로 처리. `lib/gomokuEngine.js`에 `getRingLevel`/`getRingBounds`/`isOutsideRing` 순수 함수 추가 (`ringStartMove` 기준 경과 수로 레벨 계산, 레벨당 좌/상은 `ceil(level/2)`, 우/하는 `floor(level/2)`씩 인셋 — 15→8은 홀수 차이라 비대칭 인셋으로 맞춤)
- `gameReducer.js`: `ringActive`/`ringStartMove` 상태 추가, `PICK_AUGMENT`에서 prison과 같은 패턴으로 즉시 발동(`pendingTarget` 없음), `CLICK_CELL`의 `isBlocked` 판정에 `isOutsideRing` 추가
- `GomokuBoard.jsx`: `ringBounds` prop 받아서 캔버스에 바깥 영역 반투명 검은 오버레이 + 주황 테두리로 표시. `app/local/page.js`/`RoomClient.jsx`에서 `getRingBounds(ringStartMove, totalStonesPlaced)`로 계산해서 넘겨줌

### 7. 신규 프리즘 2개 추가: 폭주(chaos) / 백지화(wipeout)
사용자가 "1. 증강 선택한 순간부터 완전 무작위로 돌을 둠 2. 판 뒤엎기처럼 모든 돌 삭제(증강/턴 수는 유지) 3. 철옹성이 무르기도 막아주나?" 3가지를 물어봄.

- **3번(철옹성-무르기)**: 확인해보니 이미 되고 있었음. `undo`(되돌리기)는 원래부터 `checkImmunity`를 거쳐서 철옹성/부적에 막힘([gameReducer.js:877-890](lib/gameReducer.js:877)). 지난 세션에 고친 건 collapse(붕괴)뿐이었고 undo는 처음부터 정상이었음.
- **폭주(chaos, prism)**: 발동 즉시 게임 끝까지 양쪽 모두 조작권 상실. 사용자 확정: 게임 끝날 때까지 계속 유지 / 클릭은 필요하지만 클릭 위치 무시하고 무작위 빈칸에 놓임 / 증강 액티브 능력 버튼도 전부 비활성화.
  - `gameReducer.js`: `chaosActive` 상태 추가, `PICK_AUGMENT`에서 prison/battleRing과 같은 패턴으로 즉시 발동. `CLICK_CELL` 진입부에서 `chaosActive`면 클릭한 (x,y)를 버리고 `pickRandomLegalCell(state, currentPlayer)`가 고른 무작위 합법 칸(금지구역/영구봉쇄/역병칸/링 바깥/렌주룰 금수 전부 제외)으로 치환 - 이후 로직은 그대로 재사용됨
  - UI: `canAct`에 `!chaosActive` 조건 추가해서 능력 버튼 비활성화 (AugmentPanel에 별도 손 안 댐, canAct만으로 충분)
  - **라이브 검증 완료**: 임시로 회차 등급 뽑기를 프리즘 고정으로 바꿔서 실제 브라우저에서 폭주를 뽑고, 이미 돌이 있는 칸을 클릭해도 다른 무작위 빈칸에 놓이며 정상적으로 턴/드래프트가 진행되는 것까지 확인 후 원복함
- **백지화(wipeout, prism)**: 1회 사용, 판 위 돌을 양쪽 다 전부 삭제하지만 보유 증강체와 stonesPlaced(진행도)는 그대로 유지. 사용자 확정: 기존 판 뒤엎기(boardFlip, 자기 돌만 재배치)와는 별도의 새 카드로 추가.
  - `gameReducer.js`의 `USE_ABILITY` 케이스에 boardFlip과 같은 패턴으로 추가 (`board: createEmptyBoard()`, `lastMove` 양쪽 다 null, 턴 넘어감)
  - `ONE_TIME_ABILITY_IDS`와 `AugmentPanel.jsx`의 `ACTIVE_ABILITIES`에 등록해서 사용 버튼이 뜨도록 함
  - 코드 리뷰로 검증(boardFlip/colorSwap과 구조 100% 동일), 프리즘 카드 등장 확률이 낮아 라이브 클릭 테스트까지는 못 함

### 8. 치명적 버그 2개 수정: 질풍노도 드래프트 인플레 / 거래·도박·동전던지기·잠복으로 얻은 즉시발동 증강 무효
사용자가 실제 플레이 중 발견한 버그 2개 제보:

**버그 A - 질풍노도(rush)가 증강 선택을 더 자주 봄**: `newStonesPlaced[currentPlayer] % 4 === 0`으로 증강 선택을 트리거하는데, 질풍노도의 보너스 돌(부스트 턴의 2번째 착수)도 이 카운트에 그대로 포함되고 있었음. 질풍노도는 2턴에 1번꼴로 실제 턴 수보다 더 빨리 4의 배수를 채우게 되어, 소유자가 상대보다 훨씬 자주 증강을 뽑는 스노우볼 버그였음.
- 수정: `isRushBonusStone`(=보너스 돌인지) 플래그를 만들어서, 보너스 돌일 때는 `stonesPlaced` 증가 자체를 건너뜀 (기존 `doubleMoveActive`가 이미 이 패턴을 쓰고 있었음 - 그걸 참고함). 겸사겸사 같은 원인으로 보너스 돌에 금지구역/감시탑 카운트다운까지 이중으로 깎이던 것도 같이 고침.
- **라이브 검증**: 임시로 프리즘 고정 뽑기를 켜서 한쪽에 질풍노도를 붙이고, 보너스 돌 포함 정확히 7번째 "카운트되는" 수를 뒀을 때 드래프트가 안 뜨고(수정 전이라면 여기서 떴어야 함), 8번째에서 정상적으로 뜨는 것까지 확인함

**버그 B - 거래/도박/동전 던지기/잠복으로 얻은 교도소·링 위에서 싸우자·폭주가 발동 안 함**: 교도소/링 위에서 싸우자/폭주는 "뽑는 순간 즉시 발동"하는 효과라 `PICK_AUGMENT` 케이스 안에서만 `prisonActive`/`ringActive`/`chaosActive` 플래그를 세팅하고 있었음. 그런데 거래(barter)·도박(gamble의 gamblePrism1)·동전 던지기(coinFlip)·잠복(ambush)은 전부 `ownedAugments`에 직접 카드를 꽂아 넣는 별도 경로라 이 세 카드 중 하나를 "우연히 얻어도" 카드만 보유할 뿐 실제 효과는 하나도 안 켜지는 조용한 버그였음. 사용자가 정확히 이 케이스(거래로 얻은 교도소)를 신고함.
- 수정: `activateInstantAugments(state, grantedAugments)` 공통 헬퍼를 만들어서 PICK_AUGMENT 본 흐름 + 도박 + 거래 + 동전 던지기 + 잠복(`triggerAmbushIfNeeded`) 전부 이 함수를 거치도록 통일함
- **라이브 검증**: 프리즘 풀을 교도소만 나오게 임시로 고정하고, 실제로 "거래" 카드를 뽑아서 사용 → 획득한 카드가 교도소로 표시되고, 상단에 "🔒 '교도소' 발동 중" 배너까지 정상적으로 뜨는 것을 확인함 (수정 전이었다면 카드만 보유하고 배너는 안 떴을 것)

두 수정 다 테스트용으로 잠깐 바꿔둔 회차 등급 고정/프리즘 풀 고정 코드는 검증 후 전부 원복함.

### 9. UI 개선 3개 + 먼저 보기(peek) 회차 등급 무시 버그 수정
- **총 수 표시 추가**: 턴 표시 아래에 "총 N수" 표시 (로컬/온라인 모드 둘 다). 각성(20수)처럼 총 착수 수 기반 증강을 가늠할 수 있게.
- **"먼저 보기" 설명 명확화**: "다음 선택지 카드 1장을 지금 미리 확정해서 봄" → "다음 증강 선택 때 나올 카드 1장을 지금 미리 확정하고 확인함 (그 카드가 다음 선택지 3장 중 하나로 반드시 포함됨)". 단순 미리보기가 아니라 그 카드가 다음 선택지에 확정으로 끼워진다는 걸 명시.
- **"증강체" → "증강" 전체 치환**: 게임 로직/UI/안내 메시지/홈페이지 문구 전부. 단순 문자열 치환이 아니라 조사도 맞춤("증강체는"→"증강은", "증강체를"→"증강을", "증강체와"→"증강과" 등 - "증강"은 받침이 있어서 "증강체"와 조사가 다름). `PROGRESS.md`는 과거 세션 기록이라 그대로 둠.
- **버그: 먼저 보기(peek)가 회차 등급 동기화를 무시함** - 사용자가 "먼저 보기로 프리즘/골드를 발견하면 상대는 실버 고르고 있을 때 나만 프리즘 선택지가 생기는데 의도한 거냐"고 질문, 확인해보니 진짜 버그였음. `peek`은 고르는 즉시 `drawFromPool`로 카드 1장을 확정 예약(`peekedCard`)하는데, 이 `drawFromPool`이 등급 제한 없이 전체 풀에서 뽑아서, 다음 회차가 실버로 정해져도 예약된 카드가 프리즘/골드일 수 있었음 - 세션 3에서 공들여 만든 "회차 등급 동기화"(같은 회차엔 두 플레이어 다 같은 등급 3장)를 실버 카드 하나로 무력화하는 셈이라 명백한 밸런스 버그.
  - 수정: `resolveRoundTier(state, roundIndex)` 헬퍼로 "회차 등급 확정" 로직을 `buildAugmentChoices`에서 분리하고, peek을 고를 때도 이 함수로 **자신의 다음 회차** 등급을 그 자리에서 미리 확정한 뒤(`draftTierPlan`에 고정), 그 등급 안에서만(`drawOneOfTier`) 예약 카드를 뽑도록 변경. `nextRoundIndex = Math.floor(state.stonesPlaced[player] / 4)` 공식은 기존 신탁(oracle)이 쓰던 것과 동일하게 맞춤.
  - **라이브 검증**: 회차 등급을 실버로 강제 고정한 상태에서 "먼저 보기"를 뽑고 다음 회차(8수)까지 진행 → 예약된 카드를 포함한 3장이 전부 실버로 나오는 것을 확인함 (수정 전이었다면 예약 카드가 골드/프리즘일 수 있었음)

모든 수정 사항은 프리뷰 서버에서 라이브로 검증 후 테스트용 임시 코드를 원복함.

### 10. 총 수 표시 UX 보강 + 먼저 보기 예약 카드 상시 표시 + 최후통첩 표시 추가
사용자가 "총 7수인데 흑돌 4수로 증강 선택 떴다, 뭔가 이상함"이라고 문의 - 실제로는 버그가 아니라 원래부터 있던 정상 동작(증강 선택은 "그 플레이어 본인 수" 기준 4의 배수마다 뜸, `roundIndex = stonesPlaced[player]/4 - 1`). 다만 지난 세션에 추가한 "총 N수" 표시가 이 사실을 가리지 않고 오히려 헷갈리게 만들었음.
- **총 수 표시를 "총 N수 (흑 N · 백 N)"로 개선**: 왜 이 타이밍에 뜨는지 바로 보이게 함 (로컬/온라인 둘 다)
- **"먼저 보기" 안내 메시지가 1.5초 만에 사라져서 뭘 예약했는지 못 봄** - `forbiddenMessage` 토스트에만 의존하지 말고, `AugmentPanel`에 "예약된 카드: 'OOO'"를 계속 표시하도록 함 (peekedCard가 소모될 때까지 유지). 온라인 모드에선 본인 화면에만 보이게(`myRole` 체크) 해서 상대에게 안 새게 함
- 이 김에 사용자가 별도로 질문한 "감시탑/최후통첩은 상대가 못 봐야 쓸모있지 않냐"도 같이 처리:
  - **감시탑**: 코드 확인 결과 "양쪽에 다 보임"이 원래 설계 의도임(금지구역/영구봉쇄도 다 상대에게 보이는 것과 같은 패턴 - "몰래 함정"이 아니라 "밟으면 손해니 알아서 피해라"는 지역 거부 도구). 사용자 확인 후 그대로 유지
  - **최후통첩**: 확인해보니 `ultimatumCell`이 어디에도 렌더링이 안 되고 있어서 상대는 물론 선언한 본인도 다시 확인할 방법이 없었음(의도된 숨김이 아니라 UI 누락). 사용자 확정: "나만 보이게 고침" → `GomokuBoard.jsx`에 `ultimatumCell`(내 선언, 보라 점선 사각형, 선명) / `fadedUltimatumCell`(상대 선언, 로컬 모드 전용, 흐리게) prop 추가. 온라인 모드(`RoomClient.jsx`)는 `fadedUltimatumCell`을 아예 안 넘겨서 상대 화면엔 절대 안 뜨게 함
  - **라이브 검증**: 로컬 모드에서 백돌이 최후통첩 선언 → 흑돌 차례일 땐 흐리게, 백돌 차례로 돌아오면 선명하게 바뀌는 것까지 스크린샷으로 확인함

테스트용으로 잠깐 바꿔둔 회차 등급 고정 코드는 검증 후 원복함.

### 11. 3차 피드백 5개: 제한시간 타이머 / 신규 증강 roleSwap / 포위 제거 프리즘 승급 / 재도전 흑백 교대 / 연속 배치 설명 개선
사용자가 스트리밍 중 5개 피드백을 한 번에 제시:

1. **⏱ 착수 제한시간(30초) 추가**: 매 착수마다 30초 안에 안 두면 무작위 합법 칸(증강/렌주룰 금지자리 제외)에 자동으로 착수하고 강제로 턴이 넘어감. `gameReducer.js`에 `TIMEOUT` 액션 케이스를 추가 — 기존 `pickRandomLegalCell`(폭주 증강이 쓰던 것)로 무작위 칸을 고른 뒤 `CLICK_CELL`을 그대로 재사용(자기 자신을 재귀 호출)해서 포위 제거/승리 판정/증강 선택 트리거 등 나머지 로직을 전부 동일하게 태움. UI(`app/local/page.js`, `RoomClient.jsx`)에 카운트다운 표시 + 매 착수마다(질풍노도 보너스 수, 양수겹침 2번째 수 포함) 타이머가 새로 시작되도록 `turnKey`(currentPlayer+lastMove 좌표 조합)를 effect 의존성으로 사용. 온라인 모드는 모든 클라이언트가 카운트다운을 표시하지만, 실제로 `TIMEOUT`을 발동시키는 건 "지금 차례인 사람의 클라이언트"뿐(안 그러면 여러 클라이언트가 동시에 쏴서 충돌).
   - **라이브 검증**: 제한시간을 3초로 임시로 낮춰서 실제로 자동 착수 + 턴 전환이 되는 것을 스크린샷으로 확인 후 30초로 원복함.
2. **신규 프리즘 증강 "입장 바꿔 생각하기"(roleSwap)**: 최초 구현은 순수 시각 효과(캔버스 렌더링에서 흑/백 채우기색만 뒤바꿔 그림)였으나, 세션 12에서 "실제로 서로 담당하는 돌 색이 바뀌는" 진짜 메커니즘으로 재구현됨 (자세한 내용은 "12번" 섹션 참고).
3. **포위 제거(capture) 골드 → 프리즘 승급**: 사용자가 이미 로컬에서 직접 등급을 바꿔둔 상태(uncommitted diff)였음 — 그대로 유지.
4. **재도전 성사 시 흑/백 교대**: 재도전이 둘 다 확정될 때마다 흑돌이었던 사람이 다음 판엔 백돌을 맡도록 변경. **온라인 모드 한정으로 구현** — 로컬 패스앤플레이는 애초에 "물리적 플레이어 신원"을 추적하지 않아서(양쪽 다 같은 화면/기기로 아무나 클릭 가능) 교대할 대상 자체가 없음. 온라인 모드는 `localStorage`에 저장된 "물리적 신원"(myRole, 방 재입장해도 고정)과 "이번 판에서 실제로 맡은 색"(myColor)을 분리: `gameReducer.js`의 `REQUEST_REMATCH`가 재도전 성사 시 `initialGameState()`로 리셋하면서 `colorFlipped` 플래그를 토글하고, `RoomClient.jsx`는 `toLogicalColor(myRole, colorFlipped)`로 매 렌더/핸들러마다 `myColor`를 계산해서 기존에 `myRole`로 하던 모든 게임 로직 비교(현재 턴 확인, 증강 선택 권한, 패널 표시, 보드 막힘 칸 등)를 전부 `myColor` 기준으로 바꿈. `myRole` 자체(신원, localStorage 저장값)는 그대로 유지 — claim/재입장 로직은 안 건드림.
   - **검증**: 리듀서 로직만 Node로 격리 테스트(`REQUEST_REMATCH`를 두 번 반복하며 `colorFlipped`가 `false→true→false`로 정확히 토글되는 것, `TIMEOUT`이 정상적으로 무작위 착수 후 턴을 넘기는 것 확인). 실제 두 브라우저 세션 간 온라인 E2E는 Supabase 계정/방 두 개가 필요해 라이브 테스트는 못 함 — 코드 리뷰로만 검증됨.
5. **연속 배치(adjacentLink) 설명 명확화**: 기존 "직전에 놓은 돌과 인접하게 놓으면 그 방향은 2칸으로 취급"이 애매하다는 피드백 → 실제 동작(직전 수와 같은 축 방향으로 바로 붙여 놓으면 그 방향 연결 길이에 +1 보너스, 즉 물리적으로 2개뿐이어도 3개를 이은 것처럼 판정됨)을 그대로 서술하도록 재작성.

### 12. roleSwap 진짜 메커니즘 재구현 + threatRadar(위험 감지) 리워크
사용자가 2가지를 요청:
1. **입장 바꿔 생각하기(roleSwap)를 진짜 메커니즘으로 변경**: "발동되면 흑돌이던 사람이 이제 백돌을 두고, 백돌이던 사람이 이제 흑돌을 둠. 승리 조건도 그 색을 따라 같이 바뀌지만, 렌주룰은 신원이 아니라 지금 흑돌을 두는 사람 기준으로 그대로 적용" — 세션 11에서 만든 "캔버스 렌더링만 색을 뒤바꿔 그리는" 순수 시각 효과를 폐기하고, 실제로 board에 놓이는 돌 색 자체가 바뀌도록 재구현.
2. **위험 감지(threatRadar) 리워크**: 상대가 두면 이기는 빈 칸을 강조 표시하던 것을, 그 승리를 완성해줄 상대 돌들을 실제로 선으로 이어서 보여주는 방식으로 변경.

**roleSwap 재구현 세부사항**:
- 지금까지 이 리듀서는 "신원"(player 1/2, ownedAugments/stonesPlaced/lastMove 등의 키)과 "보드 위 실제 돌 색"(1=흑/2=백)이 완전히 동일하다고 가정하고 있었음(신원 1은 항상 흑돌). roleSwap이 진짜로 작동하려면 이 둘을 분리해야 함.
- `gomokuEngine.js`에 `colorForPlayer(player, roleSwapActive)` 헬퍼 추가 — roleSwapActive가 켜져 있으면 1↔2를 뒤집는 대칭 변환 (신원→색, 색→신원 양방향에 똑같이 씀).
- `gameReducer.js`의 `CLICK_CELL`/`TIMEOUT`/`USE_ABILITY`(raid, boardFlip, undo의 맞불 체크)/`TARGET_CELL`(removeStone, relocate, plague, collapse)에서 board에 실제로 쓰거나 읽는 모든 지점을 `currentColor`/`opponentColor`/`playerColor` 등 `colorForPlayer`로 변환한 값으로 교체. 렌주룰 금수 판정도 "신원이 1인지"가 아니라 "지금 흑돌(색 1)을 두는지"로 변경. 반대로 `ownedAugments`/`checkImmunity`/`prison` 필터링 등 "소유권" 관련 로직은 전부 신원(player) 그대로 유지 — 사용자가 명시한 "렌주룰은 신원이 아니라 지금 흑돌 두는 사람 기준" 요구사항과 정확히 대응됨.
- 물량전(attrition)의 돌 개수 비교도 각 신원이 실제로 담당 중인 색 기준으로 다시 계산하도록 수정.
- UI(`app/local/page.js`, `RoomClient.jsx`): 턴 표시, "총 N수 (흑 N·백 N)"(이제 `countStones(board,1/2)`로 직접 세서 정확한 실제 색 기준 개수를 보여줌 — 신원별 `stonesPlaced`는 더 이상 색과 일치하지 않을 수 있어서 부적합해짐), 증강 패널 타이틀("⚫ 흑돌 증강"/"⚪ 백돌 증강" — 이제 신원이 아니라 `colorForPlayer`로 계산한 실제 색 기준으로 동적 결정), pendingTarget/증강 선택 안내 문구의 흑돌·백돌 라벨을 전부 실제 색 기준으로 수정. 온라인 모드는 기존 `myColor`(신원, colorFlipped 반영)에 더해 `myBoardColor`(`colorForPlayer(myColor, roleSwapActive)` — 실제 화면에 보이는 내 돌 색)를 새로 도입해서, 정체성 비교(턴 확인, 소유권 등)는 `myColor`, 화면 표시(roleLabel, 위험 감지/직감 계산에 넘기는 색 인자)는 `myBoardColor`로 분리.
- `GomokuBoard.jsx`의 `colorsInverted` prop(캔버스에서 렌더링 색만 뒤집던 옛 방식)은 완전히 제거 — 이제 board 배열 자체가 이미 올바른 색을 담고 있어서 그대로 그리면 됨.
- **검증**: Node로 리듀서만 격리 테스트 — 신원 1에게 roleSwap을 지급한 뒤, 신원 2(상대)가 두면 board에 색 1(흑)이 찍히고, 신원 1이 두면 색 2(백)가 찍히는 것을 확인(신원과 색이 정확히 뒤바뀜). `npx next build` 프로덕션 빌드로 전체 파일의 컴파일/타입 오류 없음도 확인. 다른 세션이 이 프로젝트 디렉터리에서 이미 dev 서버를 띄워둔 상태라 이번 세션에서 직접 브라우저 프리뷰로 라이브 검증은 못 했음 — 코드 리뷰 + Node 격리 테스트로만 검증됨. **다음에 이 프로젝트를 만질 때, 실제 브라우저에서 roleSwap을 뽑아 상대 턴에 내 색이 아닌 색이 놓이는지, 각 패널 타이틀이 실제 담당 색과 맞는지 직접 확인해볼 것.**

**threatRadar 리워크 세부사항**:
- `gomokuEngine.js`에 `findThreatLines(board, opponent, opponentOwnedAugmentIds, opponentLastMove)` 추가 — 기존 `findThreatCells`(승리 완성 빈 칸 좌표만 반환)와 달리, 각 빈 칸에 대해 승리를 만드는 축 방향으로 이미 놓인 돌들의 양 끝 좌표(`{x1,y1,x2,y2}`)를 반환. 다리 놓기(bridge)/연속 배치(adjacentLink) 보너스도 반영해서 방향별 실제 연결 길이를 계산.
- `GomokuBoard.jsx`는 `threatCells`(원 강조) 대신 `threatLines`(빨간 선 연결)를 받아서 그림. 직감(intuition)이 쓰는 `winCells`는 이번 요청 대상이 아니라서 기존 원 강조 방식 그대로 유지.
- **검증**: Node로 `findThreatLines`만 따로 테스트 — 가로 4목(흑, x=3~6)에서 좌우 양쪽 위협 좌표가 각각 올바른 돌 구간을 잇는 선으로 나오는 것, bridge 보유 시 중간 빈 칸(진짜 위협)과 양 끝 바깥 칸(6목이 되어버려 흑돌 승리 조건 불충족)이 checkWin과 동일한 기준으로 정확히 구분되는 것을 확인함. 위와 마찬가지로 실제 브라우저 라이브 검증은 다음 세션 숙제로 남음.

### 13. 부활 턴 버그 수정 + 영역 점령 너프 + 증강 선택 직후 즉시 사용 여부(보류)
같은 세션 이어서 사용자가 3가지 제시:

1. **부활(revive) 버그 수정**: 사용자가 "상대가 승리 조건을 만족해서 부활이 발동됐는데 상대가 다시 둠"이라고 제보 — 실제 버그였음. `CLICK_CELL`의 부활 분기(`gameReducer.js`)가 승리 수를 무효화하는 return 객체에 `currentPlayer`를 명시하지 않아서, 스프레드된 `stateAfterCapture.currentPlayer`(원래 값 = 이기려던 공격자)가 그대로 남아있었음 — 즉 방어자(부활 보유자)가 아니라 공격자가 턴을 유지해서 곧바로 한 번 더 두게 되는 버그. `currentPlayer: opponent`(부활을 쓴 방어자)를 명시적으로 추가해서 수정.
   - **검증**: Node로 리듀서만 격리 테스트 — 흑돌이 4목을 만들고 5번째 수로 이기는 상황을 세팅, 백돌에게 revive를 지급한 뒤 흑돌의 승리 수를 재현 → `gameOver: false`, `board[7][7]`이 다시 0으로 되돌아감, `currentPlayer`가 2(부활 보유자)로 바뀌는 것까지 확인함.
2. **영역 점령(territory) 너프**: 중앙 5x5(25칸) 기준 "7개 이상"이던 승리 조건을 "절반 이상"으로 요청 → `TERRITORY_WIN_COUNT`를 13(25의 절반 12.5를 올림)으로 변경. 증강 설명 문구도 갱신.
   - **검증**: Node로 `checkTerritoryWin` 단독 테스트 — 중앙 5x5에 12개만 채우면 승리 아님, 13개째부터 승리로 바뀌는 것을 확인함.
3. **증강 선택 직후 즉시 사용 가능하게 순서 변경(보류)**: "증강 선택 → 착수 가능 순서로 바꿔서, 뽑자마자 바로 쓸 수 있게 하면 밸런스적으로 어떨지" 질문 — 아직 구현 안 하고 트레이드오프만 답변함(아래 참고). 사용자 확답 대기 중.

**"증강 선택 직후 즉시 사용" 트레이드오프 메모**: 지금은 `PICK_AUGMENT` 후 `keepTurn`이 질풍노도 보너스 수 상황에서만 true라서, 보통은 카드를 고르자마자 바로 상대 턴으로 넘어감 — 방금 뽑은 액티브 능력(습격/재배치/무르기/양수겹침 등)은 내 다음 차례가 돌아올 때까지 못 씀. 이걸 "고르자마자 항상 keepTurn"으로 바꾸면, 그 텀을 없애주는 정도가 아니라 실질적으로 매 4턴마다 보너스 행동 하나(특히 양수겹침처럼 그 자리에서 또 착수까지 이어지는 카드)를 얹어주는 셈이라 스노우볼이 커짐. 아직 구현 전이니 사용자 확답 필요.

### 14. 도박(gamble) 증강 밸런스 수정 — 시청자 피드백("너무 사기") 반영
사용자가 "도박 증강 수정 - 선택 즉시 [실버2개 골드1개] 획득 또는 [45%확률로 프리즘 1개 획득] 중 하나를 고름"이라고 요청. 실패(나머지 55%) 시 결과를 되물었더니, 사용자는 구체적인 선택지 대신 "이 증강이 너무 사기라는 소리가 나왔어"라는 배경만 줌 — 기존 도박은 두 선택지 모두 100% 확정 획득이라 이름과 달리 진짜 도박이 아니었고, 그중 "프리즘 1개 확정 획득"이 실버 등급 카드 슬롯 하나로 프리즘 등급 뽑기 확률(1/6)을 완전히 건너뛰는 셈이라 특히 강력했던 게 "사기" 소리의 핵심이라고 판단함.
- **실패 시 결과는 "아무것도 못 얻음"으로 결정**: 프리즘 확정 획득이라는 가장 강력한 분기에 진짜 리스크(완전 꽝 가능성)를 넣어야 "너무 사기" 피드백에 실질적으로 대응되고, 이미 있는 동전 던지기(coinFlip, 실패 시 다음 드래프트 스킵만 당함)와도 험한 정도가 자연스럽게 구분됨. 실버3개 옵션은 대신 실버2+골드1로 살짝 상향해서 "안전하게 어느 정도 가치 확정" vs "위험하지만 대박/쪽박"이라는 선택 구도를 분명하게 만듦.
- `gomokuEngine.js`: `GAMBLE_OPTIONS`의 `gambleSilver3`를 `gambleMixed`(실버 2개+골드 1개)로 교체, `gamblePrism1` 설명에 "45%" 명시. 메인 `gamble` 카드 설명도 갱신.
- `gameReducer.js`의 `PICK_AUGMENT`: `gambleMixed`는 `drawSeveralOfTier`를 실버/골드용으로 두 번 호출(먼저 뽑은 실버를 골드 뽑기의 제외 목록에 넣어서 중복 방지), `gamblePrism1`은 `Math.random() < 0.45`로 성공 여부를 굴려서 성공 시에만 프리즘 1개를 뽑고 실패 시 빈 배열 - 이후 로직(획득 처리/메시지/드래프트 2회 스킵)은 공통으로 재사용. 실패 메시지는 "아무것도 얻었어요"가 아니라 "아무것도 못 얻었어요"로 자연스럽게 분기.
- **검증**: Node로 리듀서 격리 테스트 — `gambleMixed`가 항상 실버 2개+골드 1개(총 3장, 등급까지 확인)를 주는 것, `gamblePrism1`을 3000회 반복해서 성공률이 45% 근처(실측 45.5%)로 수렴하는 것, 성공 시엔 항상 프리즘 등급만 나오는 것까지 확인함. `next build`도 통과.

### 15. "더블 초이스" → "추가 선택" 이름 변경 + `pickRandom` 셔플 편향 버그 발견 및 수정
1. **이름 변경**: `doubleChoice` 증강의 표시 이름을 "더블 초이스"에서 "추가 선택"으로 변경 (`gomokuEngine.js`의 `AUGMENTS` 배열, 내부 id/필드명은 그대로 유지 - 코드 내 변수명/주석에 남아있는 "더블 초이스"는 문서용이라 안 건드림).
2. **버그 제보**: 사용자가 "도박에서 프리즘 나오는 게 대각선 강화가 엄청 나오고 있음"이라고 제보 → 실제로 뽑기 함수 자체의 편향 버그였음. `pickRandom(list, count)`가 `[...list].sort(() => Math.random() - 0.5)` 방식으로 셔플하고 있었는데, 이 패턴은 **균등 셔플이 아님** — JS 엔진의 정렬 알고리즘 특성상 배열 앞쪽에 있던 원소가 최종 결과에서 과다 선택되는 쪽으로 쏠림. `AUGMENTS` 배열에서 `diagBoost`(대각선 강화)가 프리즘 등급 중 배열상 가장 앞(사실상 전체 배열의 맨 앞)에 있어서 이 편향의 최대 피해자였음.
   - **원인 확인**: Node로 21개짜리 더미 풀에서 `pickRandom(pool, 1)`을 20만 번 반복 — 이론상 각 원소 4.76%씩 나와야 하는데, 맨 앞 원소(diagBoost 자리)가 **11.25%**(2배 이상)로 뽑히는 것을 직접 재현해서 확인함.
   - **수정**: `pickRandom`을 피셔-예이츠(Fisher-Yates) 셔플로 교체. 같은 21개짜리 풀로 재검증하니 맨 앞 원소가 정확히 **4.76%**로 돌아오고 전체 분포도 균등해짐.
   - **영향 범위**: `pickRandom`은 일반 드래프트 뽑기(`drawOneOfTier`/`drawSeveralOfTier`/`drawFromPool`)부터 도박·거래·동전 던지기·잠복·습격 대상 선정·판 뒤엎기 재배치 등 게임 전역의 모든 무작위 선택이 공유하는 함수라, 이번 수정 하나로 diagBoost 편향뿐 아니라 "특정 배열 앞쪽 항목이 은근히 자주 나온다"는 전반적인 뽑기 편향이 다 같이 고쳐짐.
   - **검증**: 실제 리듀서로 `gamblePrism1`을 5000회 돌려서 diagBoost 비중이 다른 프리즘 카드들과 동일하게 ~5%로 나오는 것, 도박 성공률(45%)은 그대로 유지되는 것을 확인함. `next build`도 통과.

### 16. 판 뒤엎기(boardFlip) 리밸런싱 — 1회용 완전 무작위 → 쿨다운제 완전 무작위
사용자가 "판 뒤엎기가 완전 쓰레기 증강이라는 소리 나옴, 프리즘급 파워는 유지하면서 밸런스 있게 리밸런싱해달라"고 요청. 브레인스토밍 여러 방향(부분 재배치/직접 배치/국소 랜덤/위협 자동 선택/쿨다운 전환 등)을 제시:
1. 처음엔 "모양 유지 이동"(연결 형태를 그대로 유지한 채 다른 빈 공간으로 통째로 이동)을 추천하고 구현까지 했음 — 사용자가 그 전 단계에서 "직접 배치는 오목 승리 조건(5목) 생각하면 너무 사기"라고 지적(자유 배치를 허용하면 액티브 능력만으로 5목을 직접 조립해 즉석 승리를 노릴 수 있음)했고, "모양 유지 이동"은 이미 정상 착수 과정에서만 만들어질 수 있는 형태(5목 불가능)를 그대로 옮기는 거라 그 사기 문제를 피해가는 대안이었음.
2. 하지만 사용자가 "이거 하지 말고 원래 있던 완전 무작위 방식으로 하되, 6수마다 1번 재사용 가능한 쿨다운제로 바꾸자"고 최종 결정 → **1번 구현은 전량 되돌리고**, 아래 쿨다운제로 교체함.

**최종 구현**:
- `gomokuEngine.js`: `boardFlip` 효과 자체는 원래 로직(내 돌 전부 지우고 지운 개수만큼 무작위 빈 칸에 재배치) 그대로 복원. 대신 설명 문구에서 "1회 사용"을 빼고 "재사용 대기시간(6수)"로 변경, `ONE_TIME_ABILITY_IDS`에서도 제거(더 이상 1회용이 아니라서 거래/봉인이 대상으로 삼으면 안 됨).
- `gameReducer.js`: `state.boardFlipCooldown: {1:0, 2:0}` 신규 필드 추가. `CLICK_CELL`에서 금지구역/감시탑 카운트다운과 완전히 같은 패턴(질풍노도 보너스 돌은 카운트 제외)으로 매 착수마다 1씩 감소. `USE_ABILITY`의 `boardFlip` 케이스는 `oneTimeUsed` 체크 대신 `boardFlipCooldown[player] > 0`이면 카드 소모 없이 "재사용 대기 중" 안내만 띄우고 리턴, 성공하면 `boardFlipCooldown[player]`를 6으로 리셋.
- `components/AugmentPanel.jsx`에 `cooldowns` prop 추가(`{ boardFlip: N }` 형태) — 대기 중이면 버튼이 비활성화되고 "재사용까지 N수"로 표시됨 (1회용의 "사용 완료"와 구분됨). `app/local/page.js`/`RoomClient.jsx`에서 `state.boardFlipCooldown[1|2]`를 각 패널에 전달.
- **검증**: Node로 리듀서 격리 테스트 — 사용 즉시 대기시간 6으로 세팅되고 턴이 넘어가는 것, 대기 중에 다시 쓰려고 하면 보드/쿨다운 변화 없이 안내만 뜨고 턴도 안 넘어가는 것, 본인이 정확히 6번 더 둔 뒤에 대기시간이 0이 되고 다시 정상 사용 가능한 것까지 확인함. `next build`도 통과.

### 17. 흑돌/백돌 전용 증강 시스템 신설 + 백돌 전용 카드 2개 추가 (테스트 결과 흑돌이 유리하다는 피드백 반영)
사용자가 테스트해보니 흑돌이 매우 유리하다고 제보 → "흑돌/백돌 전용 증강도 있으면 재밌겠다"는 아이디어에서 출발해서, 백돌(후공, 렌주룰 자유로움) 쪽 격차를 메꾸는 카드들을 브레인스토밍 → 그중 2개를 실제로 구현함.

**흑돌/백돌 전용 카드 인프라 신설**:
- `gomokuEngine.js`의 `AUGMENTS` 항목에 선택적 필드 `colorOnly: 1|2` 추가 (없으면 기존처럼 양쪽 다 등장).
- `gameReducer.js`의 뽑기 함수 3종(`drawFromPool`/`drawOneOfTier`/`drawSeveralOfTier`)에 `myColor` 파라미터를 추가하고 내부에서 `matchesColor(augment, myColor)`로 필터링하도록 리팩터링 - 매 호출부(일반 드래프트/리롤/먼저 보기/도박/동전 던지기/잠복/거래)에서 `colorForPlayer(player, state.roleSwapActive)`를 넘기도록 전부 수정. 신원이 아니라 **지금 실제로 그 색을 두고 있는지** 기준이라 입장 바꿔 생각하기(roleSwap) 중에도 정확하게 따라감.
- **검증**: Node로 300회씩 반복 드래프트 — 흑돌 신원은 백돌 전용 카드를 한 번도 못 받는 것, 백돌 신원은 받을 수 있는 것, roleSwap 발동 중에는 반대로 (지금 백돌을 두는 신원 쪽으로) 자격이 넘어가는 것까지 확인함.

**신규 카드 1 - 역감시 (counterWatch, 실버, 백돌 전용)**: 흑돌이 총 8수 이내에 처음 만드는 "열린 3목"(오픈스리 대신 이 표현으로 통일)을 감지하면 상대(백돌)에게 무료 카드 1장 지급. `countOpenThrees`(기존 렌주룰 3-3 판정에 쓰이던 함수)를 재사용해서 방금 놓인 흑돌 수가 열린 3목을 만드는지 판정, `잠복`과 같은 "무료 카드 지급" 패턴 재사용. 1회성이라 `oneTimeUsed[opponent].counterWatch`로 추적.
- **검증**: 흑돌이 8수 이내에 열린 3목을 만들면 카드가 지급되는 것, 같은 판에서 두 번째로 만들어도 중복 지급 안 되는 것, 8수 지난 뒤에는 아예 발동 안 하는 것을 Node로 확인함.

**신규 카드 2 - 양보 없음 (noYield, 골드, 백돌 전용)**: 총 4수 이내(사용자가 "8수는 너무 세다"고 해서 4수로 너프)에 흑돌이 둘 때마다 15% 확률로 백돌의 다음 턴에 보너스 착수 1회를 예약. `state.noYieldBonus: {1:false, 2:false}` 신규 필드로 "예약된 보너스" 여부를 추적 - 흑돌이 굴려서 성공하면 상대(백돌)의 플래그를 켜두고, 나중에 백돌이 실제로 둘 때 그 플래그를 소모해서 도미노와 같은 방식으로 한 번 더 놓게 함(질풍노도 보너스 돌과 달리 stonesPlaced/카운트다운에서 특별히 빼지 않음 - 도미노와 동일 취급). 이미 예약된 상태면 중복으로 안 쌓이게 가드.
- **검증**: Node로 4000회 반복해서 실측 발동률이 14.8%(목표 15% 근사)로 나오는 것, 4수 창 밖에서는 절대 발동 안 하는 것, 예약된 보너스가 실제로 백돌에게 한 번 더 놓게 해주고 그 직후 정상적으로 턴이 넘어가는 것, 이미 예약된 상태에서 재발동 시도해도 중복 안 되는 것까지 확인함. `next build`도 통과.
- 브레인스토밍 과정에서 나온 "동수 따라잡기"(증강 개수 차이만큼 보상) 아이디어는 사용자가 "매턴 4수마다 동시에 드래프트하니 격차가 거의 항상 0~1이라 사실상 카드 1장 고정 지급이랑 다를 게 없지 않냐"고 정확히 지적해서 폐기 - 대신 돌 개수 차이 기반으로 재설계하는 대안을 제안했지만 아직 구현 안 함(다음에 필요하면 참고).

### 18. 퀘스트 증강 시스템 신설(생존자/정복자) + 게임 시작 시 "시작 증강" 드래프트 추가
사용자가 신규 증강 브레인스토밍을 요청하면서 "퀘스트 증강"이라는 새 카테고리를 제안 - 특정 행동을 달성하면 그 카드의 등급보단 강하지만 한 단계 위 등급까지는 아닌 보상을 주는 카드. 후보 여러 개를 등급별(실버/골드)·색 밸런스별(백돌 캐치업)로 구조화해서 제시 → 사용자가 "퀘스트 못 채우면 무기한 대기(안전)로, 색 밸런스는 백돌 전용 카드로 접근"을 확정 → 후보 중 **생존자(survivor)**와 **정복자(conqueror)** 2개를 실제 구현 확정.

**퀘스트 증강 공통 설계**: `AUGMENTS` 항목에 `quest: true` 마커 필드 추가(로직에는 안 쓰이고 UI 배지용) - `AugmentPanel.jsx`에서 이 필드가 있으면 카드 이름 앞에 🎯 배지를 붙임. 퀘스트는 뽑는 즉시 손패에 들어가는 일반 카드와 동일하게 취급되고, 조건을 달성하는 순간 `oneTimeUsed[player].<questId>`로 소모 처리되면서 보상이 발동함(잠복/역감시와 같은 "조건 감지 → 자동 트리거" 패턴 재사용). 못 채워도 페널티 없이 무기한 보유(사용자가 "무기한 대기(안전)" 선택).

- **생존자(survivor, 실버)**: "내 돌이 이 판에서 처음 제거당하면 그 즉시 무료 실버 카드 2장 획득". `triggerSurvivorQuestIfNeeded(state, victim)` 헬퍼를 만들어서, 실제로 상대 돌이 사라지는 지점 5곳(포위 제거(capture)/습격(raid)/돌 제거(removeStone)/역병(plague)/붕괴(collapse))에 전부 훅을 걸었음. 특히 붕괴(collapse)는 3x3 범위 안에 내 돌(자폭)과 상대 돌(면역 체크 통과 시)이 동시에 사라질 수 있어서, 두 신원 모두에 대해 독립적으로 체크하도록 구현(양쪽 다 생존자를 갖고 있으면 한 번의 붕괴로 둘 다 발동 가능).
- **정복자(conqueror, 골드)**: "중앙 3x3(9칸)에 내 돌이 5개 이상이면 그 즉시 다음 증강 선택 등급이 프리즘으로 확정". `CLICK_CELL`에서 포위 제거/생존자 체크 직후, 매 착수마다 소유자 기준으로 중앙 3x3 점유 개수를 세서 체크(`triggerConquerorQuestIfNeeded`) - 신탁(oracle)과 동일한 `Math.floor(state.stonesPlaced[player]/4)` 공식으로 "다음 회차"의 `draftTierPlan`을 프리즘으로 덮어씀.
- 두 퀘스트 다 발동 메시지(`forbiddenMessage`)를 기존 여진/맞불 메시지와 자연스럽게 이어붙이도록 `extraQuestMessages` 배열 패턴으로 통일.
- (후속) 사용자가 퀘스트 카드 표시 형식을 정정: **카드 이름 자체**가 "퀘스트: 이름"으로 보여야 하고(원래 붙였던 🎯 배지는 제거), **설명**은 조건 한 줄 + 줄바꿈 + "보상: 내용" 2줄이면 됨(처음엔 desc 안에 이름까지 3줄로 넣었었는데 이름이 중복이라 잘못 이해한 것).
  - `gomokuEngine.js`의 `questDesc(condition, reward)` 헬퍼가 `조건\n보상: 내용` 2줄만 만듦(이름 인자 제거)
  - `AugmentSelectOverlay.jsx`의 `cardName`, `AugmentPanel.jsx`의 카드 이름 표시부에서 `augment.quest`면 `"퀘스트: " + augment.name`으로 표시(🎯 배지 로직 대체)
  - `app/globals.css`의 `.cardDesc`/`.augmentTooltip`에 넣은 `white-space: pre-line`은 그대로 유지(줄바꿈 렌더링용)
  - 앞으로 퀘스트 증강을 새로 추가할 땐 `desc`에 직접 문자열 쓰지 말고 `questDesc(condition, reward)`를 재사용할 것 - 이름은 자동으로 카드 표시부에서 붙음
  - (추가 후속) 조건 문장 어투를 "~이면"(조건문)에서 "~해야 합니다"(지시문) 스타일로 통일 - 예: "중앙 3x3(9칸)에 나의 돌을 5개 이상 둬야 합니다", "내 돌이 이 판에서 처음 제거당해야 합니다(...)". 새 퀘스트 조건 문장을 쓸 때도 이 어투를 맞출 것

**게임 시작 시 "시작 증강" 드래프트**: 사용자가 "각 플레이어가 첫 착수 전에 증강을 하나 먼저 고르고 시작했으면 좋겠다(룰은 그대로, 시작 증강만 추가)"고 요청.
- `initialGameState()`가 이제 빈 상태를 만든 직후 곧바로 `buildStartAugmentChoices(base, 1)`을 호출해서, 아무도 아직 안 뒀는데 흑돌의 증강 선택 화면이 이미 떠 있는 상태로 시작함(기존에 `augmentSelect`가 켜져 있으면 `CLICK_CELL`이 막히는 로직을 그대로 재사용해서 별도 게이트 코드 불필요).
- 흑돌이 고르면 `PICK_AUGMENT`의 맨 마지막(공통) return 지점에서 `state.augmentSelect.isStartDraft` 플래그를 보고 분기 - 정상 턴 넘김 대신 곧바로 `buildStartAugmentChoices(stateAfterPick, 2)`를 호출해서 백돌의 시작 증강 화면으로 이어붙임. 백돌까지 고르면 `augmentSelect: null, currentPlayer: 1`로 실제 게임을 시작함.
- 회차 등급 동기화(기존 `draftTierPlan`)와 별개로 `state.startDraftTier` 필드를 새로 둬서, 시작 증강도 기존 4턴마다 뜨는 회차들과 똑같이 "그 회차 3장은 양쪽 다 같은 등급"이 적용되도록 함.
- **구현 단순화 결정**: 금지구역(banZone)/영구봉쇄(permaBlock, 뽑는 즉시 칸 지정이 필요)과 도박(gamble, 뽑는 즉시 별도 선택 화면으로 다시 이어짐)은 시작 증강 뽑기 풀에서 제외함(`START_DRAFT_EXCLUDE_IDS`). 이 3장을 포함시키려면 "칸 지정/도박 세부선택 후 다음 플레이어 시작 드래프트로 체이닝"까지 별도로 구현해야 해서 복잡도 대비 이득이 적다고 판단 - 나머지 50여 개 카드는 시작 증강으로 전부 정상 등장 가능하고, 이 3장도 이후 4턴마다 뜨는 정상 드래프트에서는 그대로 나옴. 사용자에게 확답은 안 받았으니, 이 3장도 시작 증강에 포함시키고 싶다면 다음에 요청할 것.
- `AugmentSelectOverlay.jsx`에 `isStartDraft` prop 추가 - 제목이 "OO 증강 선택! (N수 달성)" 대신 "OO 시작 증강 선택! (착수 전)"으로 바뀜(N수=0으로 뜨는 게 어색해서). 온라인 모드(`RoomClient.jsx`)의 "상대가 고르는 중" placeholder 문구도 마찬가지로 분기.
- **검증**: Node로 리듀서 격리 테스트(시작 드래프트 체이닝이 흑→백 순서로 정확히 이어지는지, 시작 증강 3장 등급이 항상 동기화되는지, 300회 반복해도 제외 카드가 안 나오는지, 생존자 5개 트리거 지점 전부와 붕괴의 양쪽 동시 발동까지) + `next build` 통과 + 실제 프리뷰 브라우저로 로컬 모드 처음부터 끝까지 라이브 확인(흑돌 시작 증강 선택 → 백돌 시작 증강 선택 화면으로 자동 전환 → 둘 다 고른 뒤 정상적으로 흑돌 첫 착수까지 진행되는 것을 스크린샷 없이 스냅샷으로 확인).
- **온라인 모드는 코드 리뷰만** - 로컬 모드처럼 실제 두 브라우저로 "방 만들자마자 시작 증강 화면이 뜨는지, 대기 중이던 상대에게도 정상적으로 보이는지"까지는 확인 못함. 다음에 온라인 모드를 만질 일이 있으면 확인해볼 것.

### 19. "돌 개수에 영향 주는 카드는 사기"라는 사용자 피드백 반영 - 비-스노우볼 신규 증강 7개 추가
사용자가 "돌 갯수(착수 횟수 추가/제거)에 영향을 주는 카드가 밸런스적으로 사기급"이라는 원칙을 제시 → 이 원칙에 맞춰 **착수 횟수·제거를 건드리지 않는** 카테고리(규칙 조작/정보·시야/공간 제약/카드 경제/캐치업/심리·템포)로 브레인스토밍 → 사용자가 각 아이디어를 하나씩 피드백하며 최종 7개로 확정:

- **안개(fog, 실버, 온라인 전용)**: 1회 사용, 상대 턴 기준 3턴 동안 상대 화면에서만 보드 외곽 2줄이 안 보임 (세션 20에서 사용자 피드백으로 2턴→3턴 조정). `state.fogTurnsLeft[player]`로 다른 카운트다운형 효과(금지구역 등)와 동일한 패턴으로 추적. `AUGMENTS`에 `onlineOnly: true` 필드를 신설(아래 설계 패턴 참고)해서 로컬 모드 뽑기 풀에서 항상 제외됨(로컬은 양쪽이 같은 화면을 봐서 숨길 대상이 없음). 시각 효과는 밋밋한 검은 사각형 대신 `GomokuBoard.jsx`가 `boardWrapper`로 캔버스를 감싸고 그 위에 `backdrop-filter: blur` + 옅은 radial-gradient를 얹은 안개 밴드 4장(상/하/좌/우)을 CSS 애니메이션으로 흔들리게 배치.
- **예지(foresight, 골드)**: 상대가 다음에 두면 열린 3목이 되는 빈 칸을 미리 강조 표시. `gomokuEngine.js`에 `findOpenThreeSetupCells(board, opponent)` 추가(위험 감지/직감처럼 각 빈칸에 가상으로 돌을 놔보고 `countOpenThrees`로 판정하는 단순 시뮬레이션이라 미래 예측이 아니라 현재 보드 상태만으로 계산 가능). 애초에 사용자가 제안했던 "상대 다음 수를 예측"은 실시간 PvP에선 구현 불가능(온라인은 상대가 아직 클릭 안 한 위치를 알 방법이 없고, 로컬은 같은 화면이라 의미 없음)이라 폐기하고 이 방향으로 재설계됨.
- **체크무늬(checkerboard, 프리즘, 즉시발동)**: 선택 즉시 게임 끝까지 `(x+y)`가 홀수인 칸은 양쪽 다 착수 금지(짝수 칸만 생존 - 인접 칸끼리 무조건 홀짝이 뒤바뀌는 특성상 가로/세로 5목은 원천 봉쇄되고 대각선만 유효한 승리 경로로 남음). **밸런스 위험**: 대각선 강화/일자진(대각선 승리 조건 완화)과 겹치면 "판 전체가 강제로 그 카드 소유자의 승리 경로로만 흘러가는" 일방적 폭탄 조합이 되므로, `getStageExcludeIds`를 확장해서 diagBoost/straightBoost ↔ checkerboard를 서로의 뽑기 풀에서 배타적으로 제외(이미 있던 "판에 3목 있으면 diagBoost 제외" 로직과 같은 함수에 병합). `prison`/`battleRing`/`chaos`/`roleSwap`과 같은 `activateInstantAugments` 패턴으로 즉시 발동. `GomokuBoard.jsx`가 짝수 칸마다 옅은 파란 타일을 깔아서 패턴을 한눈에 보여줌.
- **벼랑 끝(brink, 골드, 퀘스트, 반복 발동)**: 내 돌이 상대보다 3개 이상 적어질 때마다(도달할 때마다 반복) 무료 실버 카드 1장 획득. 기존 퀘스트(생존자/정복자)는 전부 1회성이었는데, 이 카드는 사용자가 "버프해달라"고 요청해서 **반복 발동형**으로 설계 - `state.brinkMilestone[player]`에 "이미 보상받은 최고 격차 단계"를 기록해두고, 격차가 새로운 3의 배수를 넘어설 때만 재발동(같은 단계 중복 지급 방지, 격차가 좁혀졌다 다시 벌어져도 이미 지나간 단계는 재지급 안 됨). `CLICK_CELL`에서 매 착수 후 양쪽 신원 모두에 대해 체크(제거 계열 효과들에는 훅을 안 걸었음 - 사용자가 "상대 돌 제거할 일이 생각보다 많지 않다"고 확인해줘서, 스노우볼형 증강(질풍노도/양수겹침 등)으로 인한 돌 개수 격차가 주요 트리거가 되도록 일반 착수 경로 위주로만 구현).
- **감정(appraisal, 골드)**: 1회 사용, 보유한 증강 중 강화 가능한 카드 하나를 선택하면 그 카드의 강화판("이름+")으로 교체됨(무작위로 다른 카드를 주는 게 아니라 **그 카드 자체가 강해짐**). 강화 대상은 `ENHANCEABLE_AUGMENT_IDS`(금지 구역·감시탑·균형·늦둥이·양보 없음·역감시·축적·벼랑 끝, 총 8개)로 한정 - **착수 횟수형 카드(양수겹침/질풍노도/도미노 등)는 의도적으로 제외**해서 이 세션의 핵심 밸런스 원칙이 감정으로 다시 뚫리지 않게 함. 각 카드는 `.enhanced` 플래그 + 숫자 하나만 살짝 세지는 방식(예: 금지구역 2턴→3턴, 감시탑 4턴→6턴, 균형 2개차→1개차, 늦둥이 16수→12수, 양보없음 15%→25%, 역감시 8수→10수, 축적 1배→2배, 벼랑끝 3개차→2개차). 아래 "카드 강화" 설계 패턴 참고.
- **파기(discard, 실버)**: 1회 사용, 보유한 증강 카드 하나(자기 자신 제외)를 버리고 같은 등급의 새 카드로 교체. 이미 다 써버린 1회용 카드를 "재활용"해서 새 기회를 얻는 용도로도 쓸 수 있음(의도한 보너스).
- **노즈도르무(nozdormu, 골드, 즉시발동)**: 선택 즉시 게임이 끝날 때까지 **양쪽 모두**의 착수 제한시간이 15초로 고정. `state.timeLimitOverride`(기본 null → 30초, 발동 시 15) 하나만 두고 `activateInstantAugments`에 등록, `app/local/page.js`/`RoomClient.jsx`의 하드코딩된 `TURN_TIME_LIMIT` 상수를 `DEFAULT_TURN_TIME_LIMIT`로 이름을 바꾸고 `turnTimeLimit = timeLimitOverride || DEFAULT_TURN_TIME_LIMIT`로 매 렌더 계산하도록 변경 - 기존 turnKey 기반 타이머 리셋 로직이 그대로 재사용되어 별도 배선 불필요.
- **파기/감정의 UI 패턴**: 기존 pendingTarget은 전부 "보드 칸을 클릭"해서 대상을 고르는 구조였는데, 이 두 카드는 "내가 보유한 카드 자체"가 대상이라 새로운 액션 타입 `PICK_CARD_TARGET`(augmentId)을 추가하고, `AugmentPanel.jsx`에 `cardTargetActive`/`eligibleCardIds`/`onPickCardTarget` prop을 신설해서 패널의 카드 `<li>`를 클릭하면 그 카드가 대상으로 선택되도록 함(대상이 될 수 없는 카드는 흐리게, 될 수 있는 카드는 초록 테두리로 강조). 보드 쪽 `pendingTargetBanner`에는 "내 패널에서 선택하세요" 안내만 뜨고 실제 보드 클릭(`TARGET_CELL`)은 이 kind들에 대해선 그냥 무시됨(no-op).
- **검증(1차)**: 7개 카드 전부 Node로 리듀서/엔진 함수 격리 테스트(총 23개 assertion - 노즈도르무 즉시발동, 벼랑끝 반복발동+중복방지, 파기의 카드 교체+자기자신 선택 거부, 감정의 강화+이름 변경, 예지의 정확한 좌표 탐지, 안개의 로컬/온라인 풀 필터링 30/200회 반복, 체크무늬의 홀짝 판정+실제 착수 거부+diagBoost 배타 제외) 전부 통과. `next build` 통과.
- **검증(2차, 심층 라이브)**: 사용자가 "플레이 전에 최대한 버그가 없어야 한다"며 실제 브라우저 심층 검증을 요청 → `gameReducer.js`에 임시 디버그 훅 2개를 추가해서 실전 플레이로 전부 확인함: (1) 증강 선택 트리거를 4수마다 → 1수마다로 임시 완화, (2) 흑돌 드래프트 회차마다 정해진 카드 하나를 슬롯0에 강제로 꽂는 `debugForceChoice(player, drawn, queueIndex)` (React StrictMode의 개발 모드 이중 호출 때문에 "호출 횟수" 대신 "회차 인덱스" 기준으로 결정되게 설계 - 처음엔 mutable 큐를 shift()하는 방식으로 짰다가 이중 호출로 꼬여서 재설계함). 로컬 모드 한 판을 실제로 끝까지 플레이하며 8개 카드(균형 포함) 전부 실제로 뽑고 사용:
  - 감정으로 균형→균형+ 강화(문구가 "1개(강화)"로 실제 바뀌는 것까지 확인), 파기로 이미 써버린 감정 카드를 새 골드 카드로 교체, 노즈도르무 발동 시 배너와 함께 타이머가 실제로 15초 상한으로 도는 것, 체크무늬 발동 시 배너 + 홀수 칸 착수가 실제로 거부되고 짝수 칸은 정상 착수되는 것 + 시간초과 자동 착수(`pickRandomLegalCell`)도 체크무늬를 지키는 것까지 전부 실제 플레이로 확인.
  - **이 과정에서 진짜 버그를 하나 발견함**: 안개(fog) 오버레이의 `backdrop-filter: blur(5px)`가 `app/globals.css`에 CSS로 넣으면 이 프로젝트의 Tailwind v4 + Lightning CSS 빌드 파이프라인이 컴파일 단계에서 조용히 통째로 제거해버림 (완전히 새 `.debugBackdropTest { backdrop-filter: blur(5px); }` 규칙만 놓고 테스트해도 컴파일된 스타일시트에서 규칙째로 사라짐 - `<style>` 태그로 런타임에 직접 주입하면 정상 작동하는 것도 확인해서 빌드 도구 문제로 특정함). 원인 규명 대신 실용적으로 우회: `backdrop-filter`를 CSS 클래스에서 빼고 `GomokuBoard.jsx`의 인라인 `style` 속성으로 직접 줌(인라인 스타일은 이 빌드 파이프라인을 안 거쳐서 안전). 수정 후 실제로 상대(백돌) 턴에 `getComputedStyle`로 `backdrop-filter: blur(5px)`가 정상 적용되는 것까지 확인.
  - 예지/벼랑끝은 카드 자체가 정상적으로 뽑히고 패널에 올바르게 표시되는 것(벼랑끝은 "퀘스트: 벼랑 끝" 배지 포함)까지 확인 - 예지의 실제 캔버스 하이라이트 렌더링과 벼랑끝의 실제 격차-3 발동 트리거는 이번 라이브 패스에서 인위적으로 만들기엔 수순이 많이 필요해 스킵하고 Node 테스트 + 기존 threatRadar/winCells와 동일한 렌더링 패턴 재사용에 기댐(둘 다 새 캔버스 그리기 코드 자체가 버그 나면 콘솔 에러로 드러났을 텐데 그런 에러 없었음).
  - 검증에 쓴 디버그 훅 2개는 전부 원복 완료(원복 후 Node 테스트로 `%4` 트리거가 정상 복원됐는지 재확인, `next build` 재통과).
- **여전히 다음에 확인하면 좋은 것**: 안개 오버레이가 온라인 모드에서 실제 두 브라우저(피해자 화면에만 보이는지) 간에도 잘 격리되는지, 예지의 실제 캔버스 하이라이트 모양, 벼랑끝의 실제 격차-3 발동.

### 20. 세션 19 배포 후 사용자 실플레이 피드백 4건 반영
사용자가 바탕화면의 `증강_오목_증강목록.docx`에도 신규 증강 7개를 반영해달라고 요청 → `.docx`를 unpack해서 각 등급 표에 새 행 7개를 기존 행과 완전히 같은 서식(색상/테두리/폭, 퀘스트·전용 카드의 굵은 접두사 패턴)으로 추가하고 제목/섹션 카운트(58→65 등)도 갱신, repack 검증까지 통과.

이후 온라인 모드로 실제 플레이해보고 피드백 3개 추가 제기:
1. **프리즘 등급 카드의 패널 왼쪽 테두리 색이 골드와 구분이 안 감**: 확인해보니 실제 버그였음 - 카드 이름 텍스트는 무지개 그라데이션(`linear-gradient(90deg, #ff6b6b, #feca57, #48dbfb, #ff9ff3)`)을 쓰는데, 왼쪽 테두리는 그 그라데이션의 한 색상(`#feca57`, 노란빛)만 단색으로 쓰고 있어서 골드 등급 색(`#e0b23c`)과 육안 구분이 거의 안 됐음. `app/globals.css`의 `.augmentPanel li.tier-prism`을 `border-left-color: transparent`로 바꾸고, `::before` pseudo-element로 이름과 같은 무지개 그라데이션 3px 바를 왼쪽에 얹어서 확실히 구분되게 수정.
2. **안개 효과가 너무 옅어서 거의 안 보임**: 실제로 온라인 모드에서 안개를 뽑아 써보니, 흰색 하이라이트(0.16~0.22 알파) + 밑칠(0.32 알파)이 밝은 보드 배경(#d9a066) 위에서 거의 안 보이는 수준이었음(사용자가 "돌이 다 보인다"고 스크린샷 첨부로 제보). `.fogBand`의 배경을 90%대 알파의 짙은 회색 레이어 3장 + 짙은 밑칠로 전면 교체해서 실제로 그 밑의 돌이 안 보일 만큼 불투명하게 수정.
3. **안개 지속시간을 2턴에서 3턴으로 상향**: `gameReducer.js`의 `fogTurnsLeft[opponent]` 초기값을 2→3으로, 관련 설명 문구(`gomokuEngine.js`의 AUGMENTS desc, 발동 메시지, `AugmentPanel.jsx`의 능력 버튼 라벨) 전부 동기화.
4. **증강 리롤 시 한 번씩 버벅이는 현상**: 원인 특정 완료 - `AugmentSelectOverlay.jsx`가 카드 `<div>`의 `key`를 `augment.id + "-" + index`로 쓰고 있어서, 리롤로 그 슬롯의 `augment.id`가 바뀌면 React가 완전히 새 엘리먼트로 인식해 리마운트시킴 → `.augmentCard`에 걸린 `cardPopIn`(회전 -12deg + 확대 0.2→1.08 + 낙하 50px 오버슈트) 진입 애니메이션이 그 카드 한 장에서만 재생되면서 다른 카드는 가만히 있는데 혼자 튀는 것처럼 보였음. `key`를 카드 정체성이 아니라 슬롯 위치(`index`) 기준으로 바꿔서 리롤 시 같은 DOM 노드를 재사용하도록 수정(리마운트 안 되니 애니메이션도 재생 안 됨). 브라우저에서 리롤 전/후 DOM 노드 참조가 동일하게 유지되는 것(`dataset` 마커로 확인)까지 검증.
- **검증**: 4건 다 `next build` 통과. 1번(테두리)은 가짜 `li.tier-prism` 엘리먼트를 주입해 `::before`의 `background-image`/`border-left-color` computed style로 확인. 2번(안개 농도)은 CSS 값 변경 자체가 명확해서(0.16~0.32 → 0.9~0.95 알파) computed style 확인으로 충분히 판단, 실제 시각 스크린샷은 이 세션의 브라우저 툴 자체가 계속 타임아웃 나서 못 찍음(콘솔/네트워크/JS 실행 등 텍스트 기반 도구는 전부 정상 작동). 4번은 리롤 전후 DOM 노드 동일성을 직접 확인해서 실제로 리마운트가 없어졌음을 증명.

### 21. 증강 상호작용 버그 사냥 세션 - 코드 감사로 4건 발견 및 수정
사용자가 "오늘은 증강에 의해 나올 수 있는 버그들을 찾아내고 잡아보자"고 요청 - 새 기능 추가 없이 `lib/gomokuEngine.js`/`lib/gameReducer.js`/UI 3개 파일을 처음부터 끝까지 정독하며 증강 간 상호작용 버그를 찾는 세션. 라이브 브라우저 대신 Node로 `lib/gameReducer.js`/`gomokuEngine.js`를 직접 import하는 격리 재현 스크립트를 매 건마다 먼저 작성해 버그를 실제로 재현한 뒤 수정하고, 재현 스크립트를 다시 돌려 수정을 검증하는 방식으로 진행함(재현 스크립트는 스크래치패드에만 있고 저장소에는 커밋 안 함).

1. **[치명적] 교도소(prison)가 이미 활성화된 링 위에서 싸우자/폭주/입장 바꿔 생각하기/체크무늬를 실제로는 꺼주지 못함**: 교도소 설명은 "자기 자신 포함 모든 프리즘 효과가 비활성화"인데, 실제 구현은 프리즘 카드 4개(각각 `ringActive`/`chaosActive`/`roleSwapActive`/`checkerboardActive`라는 별도 전역 플래그로 게임플레이를 직접 좌우함) 중 어느 것도 `getActiveAugmentIds`(교도소가 프리즘을 걸러내는 유일한 지점)를 거치지 않아서, 교도소를 나중에 뽑아도 이미 켜져 있던 이 네 효과가 계속 작동하고 있었음(영구 봉쇄만 `isBlocked` 체크에 `!state.prisonActive` 조건이 이미 있어서 예외적으로 정상 작동 중이었음 - 세션 5에서 그 카드 하나만 패치했던 것). Node 재현으로 4개 전부 확인(체크무늬 홀수칸 여전히 막힘, 폭주가 여전히 클릭 위치를 무작위로 바꿈, 입장 바꿔 생각하기가 여전히 돌 색을 뒤집음, 링이 여전히 경계 밖을 막음).
   - 수정: `gameReducer.js`의 `activateInstantAugments`에서 "prison" 분기가 활성화되는 순간 이미 켜져 있던 저 4개 플래그를 직접 꺼버리도록 추가(`ringActive`/`ringStartMove`/`ringTarget`을 null로, `chaosActive`/`roleSwapActive`/`checkerboardActive`를 false로) - UI(배너, 링 경계선, 체크무늬 타일)는 이 플래그들을 그대로 읽어 그리므로 별도 수정 없이 자동으로 따라옴. 추가로 이 4개 분기 자체에도 `!state.prisonActive` 가드를 걸어서, 교도소가 이미 켜진 뒤에 이 카드들을 새로 뽑아도(카드는 손에 들어오지만) 아예 발동하지 않도록 막음(반대 순서 - 교도소 먼저, 프리즘 카드 나중 - 도 같이 커버).
   - **검증**: Node로 양방향 재현 - (a) 4개 효과를 먼저 켠 뒤 교도소를 뽑으면 즉시 다 꺼지는 것(체크무늬 홀수칸 착수 성공/폭주가 클릭한 자리에 정확히 놓임/입장 바꿔 생각하기 원상복구/링 경계 사라짐), (b) 교도소를 먼저 켠 뒤 4개 카드를 뽑으면 카드는 보유하지만 플래그가 계속 false로 남아 죽은 카드가 되는 것 전부 확인.
2. **포커페이스(pokerFace)로 흑돌/백돌 전용 카드를 훔치면 새 주인에게는 영원히 안 터지는 죽은 카드가 됨**: 흑돌 전용 균형(balance)/백돌 전용 역감시(counterWatch)·양보 없음(noYield)은 발동 조건 체크가 전부 "지금 그 색을 두고 있는 신원"인지를 따지는데, 포커페이스 강탈 대상 풀이 색 제한을 전혀 안 봐서 반대 색 신원이 훔쳐가면(백돌 전용 카드를 흑돌이 강탈하는 식) 그 즉시 조건을 절대 만족 못 하는 죽은 카드가 됐음. Node로 백돌 전용 '양보 없음'을 흑돌이 포커페이스로 강탈하는 상황을 재현해서 확인.
   - 수정: `gameReducer.js`의 포커페이스 강탈 풀 필터에 `matchesColor(a, thiefColor)` 조건 추가(`thiefColor = colorForPlayer(currentPlayer, state.roleSwapActive)`로 강탈 시점 기준 색 계산) - 색이 안 맞는 카드는 애초에 강탈 후보에서 제외됨.
   - **검증**: 상대가 색 전용 카드 하나만 들고 있는 상황에서 포커페이스가 발동하면 "강탈할 카드가 없었어요" 메시지와 함께 아무것도 안 훔쳐가는 것, 상대가 색 전용 카드 + 일반 카드를 같이 들고 있으면 일반 카드만 훔쳐가는 것을 Node로 확인.
3. **영구 봉쇄(permaBlock) 칸이 교도소 발동 후에도 화면에는 계속 막힌 것처럼 표시됨**: 세션 5에서 `gameReducer.js`의 실제 착수 판정(`isBlocked`)에는 `!state.prisonActive` 조건을 넣어서 기능적으로는 이미 풀려 있었지만, `app/local/page.js`/`components/RoomClient.jsx`가 보드에 X 표시를 그릴 칸 목록(`boardBlockedCells`/`fadedBlockedCells`)을 계산할 때는 이 조건을 안 넣어서, 실제로는 둘 수 있는 칸에 여전히 막힌 것처럼 X가 남아있는 시각적 불일치였음(기능은 정상, 표시만 안 따라옴).
   - 수정: 두 파일 모두 `prisonActive`가 켜져 있으면 `permaBlockedCells`를 빈 배열 취급하도록 변경.
4. **체크무늬(checkerboard)와 각성(awakening)을 같이 뽑으면 체크무늬 vs 대각선강화/일자진 배타 처리가 우회됨**: 세션 19에서 체크무늬가 대각선 강화/일자진(둘 다 대각선 승리 조건 완화)과 겹치면 폭탄 조합이 된다고 판단해 서로 뽑기 풀에서 배타적으로 제외하도록 만들어뒀는데, 각성(20수 이후 자동으로 대각선강화+일자진과 완전히 같은 효과를 얹어주는 카드)은 이 배타 목록에서 빠져 있었음 - 체크무늬+각성 조합으로 20수만 넘기면 정확히 같은 폭탄 조합이 그대로 재현됐음.
   - 수정: `gameReducer.js`의 `getStageExcludeIds`에서 각성을 diagBoost/straightBoost와 동급으로 취급하도록 양방향 추가(각성 보유 시 체크무늬 제외 풀에 추가 / 체크무늬 활성 시 각성도 숨김 목록에 추가).
   - **검증**: Node로 각성 보유 시 프리즘 300회 리롤에서 체크무늬가 한 번도 안 나오는 것, 체크무늬 활성 시 프리즘 300회 리롤에서 각성이 한 번도 안 나오는 것 확인.

네 건 모두 `next build` 통과 확인(1건은 이 세션에서 `npm install`로 처음 `node_modules`를 설치해야 했음 - 설치 과정에서 `package-lock.json`이 의도치 않게 변경돼서 커밋 전에 `git checkout`으로 되돌림). 온라인 모드는 코드 리뷰 + Node 리듀서 테스트로만 검증(실제 두 브라우저 라이브 확인은 못 함 - 특히 1번 교도소 버그는 온라인 대전에서 더 자주 부딪힐 만한 조합이라 다음에 온라인 모드를 만질 일이 있으면 우선 확인해볼 것).

사용자가 "다른 버그는 없어?"라고 이어서 질문 - 같은 방식(Node 재현 스크립트 먼저, 그다음 수정)으로 4건 더 발견·수정:

5. **최후통첩(ultimatum)이 발동(fulfill)된 뒤에도 `ultimatumCell`이 안 지워짐 - 그 칸의 돌이 나중에 사라졌다가 같은 자리에 다시 놓이면 다리 놓기+연속 배치 보너스가 무한히 재발동됨**: 카드 설명 자체가 "내가 그 칸에 처음 두는 순간"이라고 명시하는데, 실제 구현은 `ultimatumCell[currentPlayer]`를 선언 시점에 한 번 세팅한 뒤 발동 여부만 매 수마다 좌표 일치로 검사하고 발동 후 초기화하는 코드가 어디에도 없었음. 그 칸의 돌이 스스로 쓴 직전 무르기(selfUndo)·상대의 되돌리기(undo)·돌 제거·포위 제거 등으로 다시 비워지고, 그 자리에 다시 두면 매번 보너스가 재적용됨. Node로 "선언 → 이행 → 직전 무르기로 그 수만 취소 → 같은 자리에 재착수" 시퀀스를 재현해서 두 번째 이행에도 `ultimatumCell`이 그대로 남아있는 것을 확인.
   - 수정: `gameReducer.js`의 `CLICK_CELL`에서 `ultimatumFulfilled`가 true인 바로 그 수에 `stateAfterCapture.ultimatumCell[currentPlayer]`를 `null`로 지우도록 추가(캡처/퀘스트 트리거보다 앞서, 이후의 모든 return 경로(승리/보드가득참/양수겹침 계속 등)가 전부 이 `stateAfterCapture`를 기반으로 하므로 한 곳만 고치면 전체 경로에 반영됨).
   - **검증**: 위 재현 시퀀스를 다시 돌려서 두 번째 이행 시도에서 `ultimatumCell`이 이미 `null`이라 보너스가 다시 붙지 않는 것 확인.
6. **재도전(rematch) 화면의 "⚫ 흑돌"/"⚪ 백돌" 버튼 라벨이 입장 바꿔 생각하기 발동 중엔 실제 담당 색과 반대로 표시될 수 있음**: `WinOverlay.jsx`가 신원(player 1/2)을 항상 "1=흑돌, 2=백돌"로 고정 라벨링하고 있었는데, 게임 도중 입장 바꿔 생각하기가 발동해 있었다면 신원과 실제 담당 색이 뒤바뀐 상태로 게임이 끝났을 수 있어서 라벨이 실제와 반대로 보일 수 있었음(기능 자체는 정상 - 버튼을 누르면 정확히 그 신원의 재도전 요청이 가는 건 맞지만, 아이콘/텍스트가 헷갈림). 세션 12에서 roleSwap을 위해 `myBoardColor` 개념까지 만들어 다른 곳은 다 이 패턴을 따랐는데 `WinOverlay`만 빠져 있었던 것으로 보임.
   - 수정: `WinOverlay.jsx`에 `roleSwapActive` prop을 추가하고 `colorForPlayer(player, roleSwapActive)`로 라벨을 동적으로 계산하도록 변경. `app/local/page.js`/`components/RoomClient.jsx` 양쪽 호출부에서 `roleSwapActive`를 넘기도록 수정.
7. **도박(gamble)의 "프리즘 1개 획득(45%)" 옵션이 영구 봉쇄(permaBlock)를 뽑아줄 수 있는데, 그러면 칸을 지정할 방법이 없어 영원히 죽은 카드가 됨**: 금지 구역/영구 봉쇄는 "뽑는 즉시 칸을 지정"해야만 효과가 생기는 카드라(`PICK_AUGMENT`의 전용 분기에서만 `pendingTarget`을 만듦), 이 두 카드를 정상 드래프트 이외의 경로로 손에 쥐여주려면 반드시 `IMMEDIATE_TARGET_EXCLUDE_IDS`로 걸러야 한다는 원칙이 이미 코드 곳곳에 있었는데(동전 던지기/잠복/역감시/파기/거래의 실버·골드 뽑기 등), `gamblePrism1`(프리즘 전용 뽑기)만 이 필터가 빠져 있었음 - 영구 봉쇄가 프리즘 등급이라 실제로 걸릴 수 있었음. Node로 3000회 반복 중 25회 만에 실제로 영구 봉쇄를 뽑아서 `pendingTarget`이 `null`인 채로 카드만 손에 들어오는 것을 확인.
   - 수정: `gamblePrism1`의 뽑기 풀에도 `IMMEDIATE_TARGET_EXCLUDE_IDS`를 hardExclude에 추가.
8. **거래(barter)도 같은 이유로 영구 봉쇄를 뽑아줄 수 있음**: 7번과 완전히 같은 원인 - `barter`가 프리즘 카드를 고를 때 `AUGMENTS`를 직접 필터링하면서 이 배제 목록을 아예 안 거쳤음. Node로 3000회 중 8회 만에 재현.
   - 수정: `barter`의 `prismPool` 필터에도 `!IMMEDIATE_TARGET_EXCLUDE_IDS.includes(a.id)` 조건 추가.
   - (7, 8 두 건 모두 확인 차원에서 코드 내 다른 모든 카드 지급 경로 - 잠복/생존자/벼랑끝/일반 드래프트/시작 드래프트/역감시/동전 던지기/파기/먼저 보기/리롤 - 를 전수 점검해서 이 두 곳 외에는 전부 이미 올바르게 처리돼 있는 것을 확인함. 일반 드래프트/시작 드래프트/먼저 보기/리롤은 배제할 필요가 없음 - 그 카드들은 정상적인 `PICK_AUGMENT` 흐름을 그대로 타서 고르는 순간 정상적으로 `pendingTarget`이 걸림.)

8건 모두 Node 재현 스크립트로 버그 확인 → 수정 → 같은 스크립트로 재검증 → 전체 스크립트 묶어서 재실행 → `next build` 순서로 검증 완료.

사용자가 "더 찾아보자, 이번엔 꼼꼼하게"라고 재요청 - 명백한 로직 오류는 더 안 나왔지만, "둘 다 뽑으면 하나가 죽어버리는" 카드 상호작용 2건을 찾아 보고함(바로 고치지 않고 먼저 사용자 판단을 구함 - 정답이 하나로 정해지지 않는 밸런스/설계 문제라서):
- **포위 제거(capture) + 오델로(othello)를 같이 보유하면 오델로가 항상 죽은 카드가 됨**: 둘 다 같은 "나-상대-나" 패턴을 감지하는데, 코드 순서상 capture가 먼저 그 칸을 제거해버려서 othello의 `findCaptures` 재탐색 시점엔 이미 빈 칸이라 뒤집을 대상이 없어짐. Node로 두 카드를 함께 가진 상태에서 재현해서 항상 제거만 일어나고 뒤집기는 한 번도 발동 안 하는 것 확인. (아직 미해결 - 사용자에게 "배타적으로 뽑기 풀에서 제외 / 무작위로 어느 효과 적용할지 결정 / capture 우선으로 명시" 중 선택지를 물어봄)
- **질풍노도/양수겹침의 보너스 착수가 링 위에서 싸우자의 축소 시계(4수마다 한 겹)를 안 돌림**: "질풍노도 드래프트 인플레" 버그(세션 8)를 고치면서 보너스 착수를 `stonesPlaced`에서 빼도록 했는데, 링의 축소 타이머도 이 `stonesPlaced` 합계를 그대로 재사용하고 있어서 그 두 카드를 보유/사용 중인 턴엔 실제로 돌이 2개 놓여도 링의 "경과 수"는 1만 올라가는 것을 확인.

사용자가 이어서 "링 위에서 싸우자 발동 시 두 플레이어의 모든 증강을 비활성화하는 건 어떠냐"고 제안 - 그건 카드의 정체성 자체를 바꾸는 큰 변경(사실상 "교도소 + 좁아지는 판"이 되어 교도소와 정체성이 겹치고 fortress/revive/capture/othello 등이 링 발동 중 전부 죽어버림)이라고 반대 의견을 제시하고, 대신 "링의 축소 타이머가 실제로 보드에 놓인 돌 총 개수를 직접 세도록" 하는 더 수술적인 대안을 역제안 - 사용자가 그 방향으로 승인.

### 21-1. 링 위에서 싸우자 축소 시계를 `stonesPlaced` 대신 전용 카운터(`placementClock`)로 분리
막상 구현하려고 보니, 원래 제안대로 "보드 위 돌 총 개수"를 그대로 쓰면 포위 제거/습격/무르기 등으로 돌이 사라질 때마다 `elapsed`가 줄어들어 **이미 좁아진 링이 도로 넓어지는 새 버그**가 생긴다는 걸 발견함(무르기/직전 무르기는 기존에도 `stonesPlaced`를 감소시켜서 이미 조금 있던 문제였지만, 포위 제거·습격·역병·붕괴·돌 제거처럼 훨씬 자주 쓰이는 카드들까지 전부 이 "링 역주행" 경로를 타게 되어 체감 빈도가 크게 늘어날 것으로 판단). 그래서 원안 대신 **절대 감소하지 않는 전용 카운터**로 구현:
- `gameReducer.js`에 `state.placementClock`(0부터 시작, 신원 구분 없는 단일 숫자) 신규 필드 추가. `CLICK_CELL`에서 `stateAfterCapture`가 만들어진 직후(캡처/퀘스트/최후통첩 처리보다 먼저) 무조건 `+1` - 질풍노도 보너스 착수든 양수겹침 둘째 착수든 실제로 보드에 새 돌이 놓이는 모든 경로가 이 한 지점을 반드시 거치므로 별도 분기 없이 전부 커버됨. 부활로 이 수의 승리가 나중에 무효화되더라도(그 반환 경로도 `stateAfterCapture`를 스프레드하므로) 이미 지나간 시간이라 되돌리지 않음 - 포위 제거/퀘스트 지급 등 revive가 이미 기존에도 되돌리지 않던 다른 부수효과들과 동일한 패턴.
- 링 관련 계산 4곳(`activateInstantAugments`의 `ringStartMove` 초기화, `CLICK_CELL`의 착수 전/후 `ringBounds`·`newRingBounds`, `pickRandomLegalCell`)과 UI 2곳(`app/local/page.js`, `RoomClient.jsx`)에서 전부 `state.stonesPlaced[1]+state.stonesPlaced[2]` 대신 `state.placementClock`을 쓰도록 교체. 각성(awakening)의 20수 임계값·인색/늦둥이 등 링과 무관한 다른 `stonesPlaced` 참조는 그대로 유지(성급하게 전부 바꾸면 각성 타이밍이 어긋나므로 변수명 구분에 주의).
- **검증**: Node로 (1) 질풍노도 부스트 턴 2개 착수 모두 `placementClock`을 올리지만 `stonesPlaced`는 1만 오르는 것, (2) 양수겹침 두 착수 모두 마찬가지, (3) 돌 제거를 써도 `placementClock`이 그대로인 것(감소 안 함), (4) 링 발동 시 `ringStartMove`가 그 시점의 `placementClock`과 일치하는 것 확인. 마지막으로 질풍노도+링 위에서 싸우자를 함께 가진 시나리오로 실제 착수 3번(부스트 턴 2개+상대 1개)까지는 링이 안 좁혀지고 정확히 4번째 실제 착수에서 레벨 1로 좁혀지는 것까지 end-to-end로 확인(수정 전이었다면 이 타이밍에 아직 레벨 0이었을 것). `next build` 통과.
- **아직 안 고친 것**: capture+othello 동시 보유 시 othello가 죽는 문제는 사용자 확답 대기 중.

### 21-2. 포위 제거 + 오델로 동시 보유 시 오델로가 죽는 문제 - "50% 확률 절충"으로 해결
사용자가 "밸런스적으로 딱 좋게 해봐"라고 요청 - 세 가지 선택지(배타적 뽑기 제외 / 무작위 절충 / capture 우선 명시) 중 무작위 절충을 선택함. 뽑기 풀 배제는 두 카드가 테마상 가장 가까운 짝(둘 다 "나-상대-나" 패턴을 노림)인데 아예 같이 못 뽑게 막는 게 아쉽고, capture 우선 명시는 그냥 죽은 카드를 문서화만 하는 것뿐이라 실질적 밸런스 개선이 안 됨 - 무작위 절충이 둘 다 값어치를 유지하면서 "낀 돌마다 어느 효과가 나올지 모르는" 재미있는 불확실성도 더해서 가장 나은 선택으로 판단.
- `gameReducer.js`의 캡처 처리 블록을 `findCaptures`를 한 번만 호출하는 단일 루프로 재구성 - 포위 제거만 있으면 항상 제거(기존과 동일), 오델로만 있으면 항상 뒤집기(기존과 동일), **둘 다 있으면 낀 돌 하나하나마다 50% 확률로 제거/뒤집기 중 하나를 결정**(`applyCapture = hasCapture && (!hasOthello || Math.random() < 0.5)`). 여진의 제거 방어는 "제거" 결과가 나왔을 때만 체크(기존 설계상 여진은 애초에 "뒤집기"는 안 막아줌 - 그대로 유지), 도미노 보너스 턴은 두 결과 중 어느 쪽이든 `capturedOrFlippedAny`로 동일하게 트리거, 생존자 퀘스트는 "제거" 결과일 때만(`captureRemovedAny`) 트리거되도록 기존 구분을 그대로 유지.
- `AUGMENTS`의 포위 제거/오델로 설명 문구에 "함께 보유 중이면 낀 돌마다 50% 확률로 둘 중 하나가 적용됨"을 추가.
- **검증**: Node로 (1) 둘 다 보유 시 2000회 반복해서 제거/뒤집기가 각각 ~50%(실측 50.1%/49.9%)로 나오고 둘 다 절대 놓치지 않는 것, (2) 포위 제거만 보유 시 항상 제거(기존 동작 그대로), 오델로만 보유 시 항상 뒤집기(기존 동작 그대로), (3) 상대가 철옹성이면 둘 다 여전히 완전히 막히는 것, (4) 도미노가 두 결과 중 어느 쪽이든 항상 보너스 턴을 주는 것(30/30), (5) 여진이 "제거" 결과만 막아주고 "뒤집기" 결과는 안 막는 것(기존 설계 그대로 유지됨을 재확인), (6) 생존자 퀘스트가 "제거" 결과에서만 발동하고 "뒤집기" 결과에서는 발동 안 하는 것까지 전부 확인. `next build` 통과.

(참고: 이 버그 사냥 세션이 진행되는 동안 별도 세션이 master에 직접 신규 프리즘 증강 2개(시공간 붕괴/결계, 아래 22번 섹션)를 커밋함 - PR을 master에 합치면서 두 작업 내역을 이 문서에도 같이 정리함. `activateInstantAugments`에 `player` 인자가 추가되는 등 겹치는 부분이 있었지만 실제 코드 충돌은 없었고, 이 문서의 서로 다른 "21번" 섹션 번호만 정리해서 합침.)

### 22. 신규 프리즘 증강 2개 추가: 시공간 붕괴(timeCollapse) / 결계(ward)
사용자가 "프리즘 증강 아이디어 생각해보자"고 요청 → 8개 아이디어(역린/결계/잔상/연쇄반응/거울/각인/황혼/분신) 제시 → 사용자가 "잔상"(이름을 "시공간 붕괴"로 확정)과 "결계"를 선택. 결계는 원안이 "내 돌 2개를 지정해 그 사이 직선을 상대만 착수 금지"였는데, 사용자가 "대각선으로 길게 그으면 판 절반을 일방적으로 봉쇄할 수 있어 너무 셀 것 같다"고 우려 → 논의 끝에 **양쪽 모두에게 적용되는 대칭형 장벽**으로 확정(한쪽만 막는 함정이 아니라 둘 다 못 넘는 지형 조작 카드로 재설계).

- **시공간 붕괴 (timeCollapse, 프리즘)**: 획득하는 순간 그 시점의 보드를 조용히 저장해두고, 게임 중 아무 때나 1회 그 시점으로 판을 되돌림(양쪽 돌 전부 대상, `stonesPlaced`/보유 증강은 그대로 유지, 사용하면 턴 넘어감). boardFlip/wipeout/colorSwap과 같은 "버튼 하나로 즉시 발동, 사용하면 턴 넘어감" 패턴 재사용.
  - **획득 시점 스냅샷을 모든 획득 경로에서 빠짐없이 찍어야 하는 문제**: 세션 8의 버그(거래/도박/동전 던지기/잠복으로 얻은 즉시발동 프리즘이 발동 안 되던 문제)와 정확히 같은 종류의 함정이라, 그때 만든 `activateInstantAugments(state, grantedAugments)` 공통 헬퍼를 재사용하기로 함 - 다만 기존 함수는 prison/battleRing/chaos/roleSwap/checkerboard/nozdormu처럼 "누가 얻었는지와 무관한 전역 효과"만 다뤄서 `player` 인자가 아예 없었음. timeCollapse는 "그 사람 개인의" 스냅샷이라 `player`가 필요해서, 함수 시그니처에 `player` 파라미터를 추가하고 이 함수를 부르는 7곳(PICK_AUGMENT 본 흐름/도박/거래/동전 던지기/잠복/역감시 보너스/파기 교체) 전부에 실제 `player`(또는 잠복의 `victim`, 역감시의 `opponent`)를 넘기도록 수정. 이 리팩터가 기존 즉시발동 카드들을 깨지 않았는지 prison/checkerboard/nozdormu로 회귀 테스트 추가.
  - **역병/결계와의 상호작용 버그를 미리 잡음**: 스냅샷 당시엔 돌이 있었는데 그 사이에 역병이나 결계로 그 칸이 "영원히 죽은 칸"(`deadCells`)이 됐다면, 되돌리기가 그 죽은 칸에 돌을 부활시켜버리는 게 아닌가 하는 우려가 있어서, 사용 시점에 `deadCells`에 포함된 칸은 스냅샷에 뭐가 있었든 무조건 다시 비우도록 방어 코드를 넣음(deadCells를 "영원히 비어야 한다"는 유일한 하드 불변식으로 보고, 링/체크무늬/금지구역/영구봉쇄처럼 "이후 착수만 금지"인 것들은 기존 코드(체크무늬 발동 시 기존 돌 안 건드림)와 같은 기준으로 굳이 안 건드림).
  - 면역(철옹성/부적) 체크는 안 함 - boardFlip/wipeout/colorSwap과 같은 "전역 리셋" 계열은 애초에 면역 체크 대상이 아니라는 기존 코드 패턴을 그대로 따름(공격이 아니라 환경 조작이라는 취지).
- **결계 (ward, 프리즘)**: 1회 사용 - 판 위의 두 칸을 지정(가로/세로/대각선 일직선이어야 함, 아무 칸이나 - 내 돌/상대 돌/빈칸 무관). 그 사이에 있는 빈 칸들이 게임 끝까지 **양쪽 모두** 착수 금지가 됨. 새 상태 필드 없이 기존 역병(plague)의 `deadCells`(영원히 양쪽 다 착수 불가) 배열을 그대로 재사용해서 구현 - 그 덕에 잠복/도장깨기/성역 같은 다른 증강들이 "역병으로 죽은 칸은 원래도 특별 취급 안 함"이라는 기존 전제를 그대로 물려받아서 새로운 상호작용 버그 표면이 거의 안 생김.
  - `banZone`/`permaBlock`처럼 "뽑는 즉시 칸 지정"하는 대신, `collapse`/`plague`/`relocate`/`removeStone`처럼 **버튼을 눌러야 시작되는 능력**으로 설계 - 그래서 `IMMEDIATE_TARGET_EXCLUDE_IDS`/`START_DRAFT_EXCLUDE_IDS`에 추가할 필요가 전혀 없었음(banZone/permaBlock이 그 목록에 있는 이유는 "즉시 칸 지정"이 거래/도박/잠복 같은 카드-only 획득 경로와 안 맞기 때문인데, 결계는 그 문제 자체가 없음).
  - 두 칸이 일직선이 아니면: 카드를 소모하지 않고 첫 번째 선택은 유지한 채 안내 메시지만 띄우고 다시 고르게 함(`removeStone`/`relocate` 등 기존 대상 선택 능력들의 "잘못 클릭하면 조용히 무시" 관례와 달리, 일직선 여부는 눈으로 바로 안 보이는 조건이라 이번엔 메시지를 띄우기로 함).
  - 사이에 이미 돌이 있는 칸은 건너뜀(죽은 칸으로 안 만듦, 기존 돌은 그대로 유지) - 나중에 그 돌이 제거되면 그 자리는 다시 자유롭게 둘 수 있음(플레이버상 약간 어색할 수 있지만, "결계 == deadCells 재사용"이라는 단순함을 지키기 위한 의도적 결정).
- **밸런스 논의**: 결계를 상대 전용(비대칭)으로 만들면 대각선으로 길게 그어 판 절반을 일방적으로 봉쇄하는 게 사실상 "그 판을 결정짓는" 카드가 될 위험이 커서, 사용자와 상의 후 대칭형으로 확정 - 나도 못 넘는 벽이라 위협적인 승리 라인을 미리 끊는 방어나 국지전으로 유도하는 지형 조작 용도로만 쓰이게 됨.
- **검증**: Node로 리듀서 격리 테스트 총 64개 assertion (스냅샷 캡처, 되돌리기 후 lastMove/oneTimeUsed/턴 처리, deadCells 불변식, roleSwap 활성 중 정상 동작, 교도소 중 사용 불가, 거래로 획득 시에도 스냅샷이 찍히는지, 결계의 가로/대각선 정렬 판정, 정렬 실패 시 카드 미소모, 결계로 막힌 칸의 양쪽 착수 거부, 인접 칸 선택 시 0칸 효과도 정상 소모, 사이에 돌이 있으면 건너뜀, activateInstantAugments 리팩터 회귀) 전부 통과 + `next build` 통과.
  - **라이브 브라우저 검증**: 세션 19-20 방식대로 `buildStartAugmentChoices`에 임시 디버그 훅을 넣어 시작 증강 화면에서 흑돌은 시공간 붕괴, 백돌은 결계를 강제로 받게 한 뒤, Playwright로 로컬 모드를 처음부터 끝까지 실제로 조작: 흑백이 각 1수씩 두고 → 흑돌이 시공간 붕괴 사용 → "총 2수 (흑 0 · 백 0)"로 판만 비워지고 진행 수는 유지되는 것 확인 → 백돌이 결계 사용 → 두 칸(가로 4칸 간격) 선택 → "3칸이 양쪽 다 착수 금지됐어요" 메시지 확인 → 흑돌이 그 막힌 칸을 클릭 → "여기는 막혀 있어서 놓을 수 없어요"로 정상 거부되는 것까지 전부 확인. 검증 후 디버그 훅은 완전히 제거하고 원본과 diff로 클린 리버트 확인, Node 테스트/`next build` 재통과.
  - 이 과정에서 `/local` 페이지의 기존(무관한) 하이드레이션 경고를 발견함 - `initialGameState()`가 렌더 중 `Math.random()`으로 시작 증강을 뽑아서 서버 렌더와 클라이언트 첫 렌더의 카드가 어긋나는 문제로, 디버그 훅을 끈 상태에서도 재현되는 걸 확인해 이번 세션 변경과 무관한 기존 이슈로 확인함(원인 파악만 하고 손은 안 댐 - 범위 밖).

### 23. 싱글플레이(컴퓨터 대전) 모드 추가 - 외부 API 없는 휴리스틱 AI
사용자가 "1대1 싱글플레이가 가능한지, 상황에 맞춰 증강까지 골라가며 두는 AI가 필요하다(앤트로픽 API는 안 씀)"고 요청 → 가능하다고 답하고 휴리스틱 기반으로 구현 확정. 사용자가 특히 "앞으로 계속 늘어나거나 패치될 증강까지 AI가 대응해야 한다"고 강조해서, "증강 id마다 AI 코드를 하드코딩"하는 대신 최대한 실제 리듀서를 **dry-run(시험 삼아 실행해보고 결과만 관찰)** 하는 방식으로 설계함 - 새 증강이 `gameReducer.js`에 표준 패턴(USE_ABILITY/TARGET_CELL/PICK_CARD_TARGET 케이스)으로 추가되기만 하면 `lib/aiPlayer.js`를 안 고쳐도 AI가 자동으로 인식하고 다룸.

**신규 파일 `lib/aiPlayer.js`** (순수 함수, DOM/React 의존 없음 - `gomokuEngine.js`/`gameReducer.js`와 같은 계층):
- **착수 선택**: ①즉시 승리 수가 있으면 최우선 ②없으면 상대의 다음 수 즉시 승리 자리를 반드시 차단 ③그 외엔 색상 무관 패턴 매칭 휴리스틱(`scanLine`으로 4방향 라인을 'X'(내 돌)/'O'(상대·벽)/'.'(빈칸) 문자열로 만들고 열린4/막힌4/열린3 등 패턴 정규식으로 점수 매김, 렌주룰 금수·링/체크무늬/금지구역 등은 `getLegalCells`로 필터링)로 후보 중 최고점 + 약간의 무작위성으로 선택.
- **증강 선택(드래프트)**: 등급 기반 기본 점수(`TIER_BASE_SCORE`: 실버10/골드22/프리즘38) + 알려진 카드에 대한 소소한 보정치(`AUGMENT_PICK_HINTS`, 모르는 id는 0이라 새 카드가 추가돼도 등급 점수만으로 무난하게 평가됨) + 퀘스트 가산점. 가장 약한 슬롯이 리롤 안 됐고 너무 약하면 1회 리롤.
- **액티브 능력 사용 여부 판단**: owned 카드 각각에 대해 실제로 `USE_ABILITY`를 dry-run 해보고 "안내 메시지만 말고 진짜로 상태가 바뀌는지"(`stateMeaningfullyChanged`)로 "지금 쓸 수 있는지"를 판정 - 쿨다운/1회용 소모/leverage 같은 조건부 자격 체크를 리듀서 게이트 그대로 재사용해서 AI 쪽에 중복 코드가 없음. "지금 쓸 가치가 있는지"만 우선순위표(`ABILITY_PRIORITY`, 상대 즉시위협 여부/판 형세/중반 여부 기준)로 판단하고, 표에 없는(미래) 능력은 기본 우선순위로 적당히 시도해봄.
- **대상(칸/카드) 선택**: `pendingTarget`/`PICK_CARD_TARGET` kind 이름을 몰라도 됨 - 후보(판 전체 칸, 또는 보유 카드 전부)를 실제로 dry-run해서 리듀서가 받아들이는(상태가 바뀌는)것들 중 결과 보드 평가(`fullBoardScore`)가 가장 좋은 걸 고름. 재배치/결계처럼 2단계 능력은 1단계 더 내다봄(깊이 1로 제한해서 성능 확보). discard는 점수를 반전시켜 "가장 덜 쓸모있는" 카드를 버림.
- 진입점 `decideAiAction(state, aiPlayer)` 하나만 export - 호출 측이 반환된 액션 1개를 dispatch하면, 그 결과로 갱신된 state로 다시 이 함수를 부르는 식으로 한 턴(질풍노도 보너스 수, 능력 여러 번 사용 등)이 여러 틱에 걸쳐 자연스럽게 진행됨. `isAiTurn(state, aiPlayer)`도 같이 export(UI가 사람 입력을 막을 시점 판단용).

**`lib/gameReducer.js` 소폭 리팩터** (AI가 재사용할 수 있게 기존 내부 함수 3개를 export로 전환 - 동작 변경 없음):
- `getActiveAugmentIds`/`checkImmunity`export 추가.
- `pickRandomLegalCell` 내부 로직을 `getLegalCells(state, player)`로 뽑아내 export(폭주의 무작위 착수와 AI의 착수 후보 계산이 정의 하나를 공유).

**버그 발견 및 수정**: AI vs AI 자동 대국으로 40판→300판 반복 테스트하다가 실제 게임 버그를 하나 발견함 - **도장깨기(jailbreak)를 막힌 자리가 하나도 없는데 사용하면 `pendingTarget`만 생기고 아무 데도 유효한 대상이 없어서 게임이 영원히 멈춤(취소 액션이 없음)**. 이건 AI뿐 아니라 사람이 실수로 도장깨기를 써도 똑같이 걸리는 진짜 소프트락 버그였음 - `discard`/`appraisal`이 이미 쓰던 "대상이 없으면 카드 소모 없이 안내만 띄움" 패턴을 그대로 적용해서 수정(`state.blockedCells`/`permaBlockedCells`/`watchtowerCells`에 이 플레이어 몫이 하나도 없으면 즉시 `forbiddenMessage`만 띄우고 리턴).

**신규 라우트 `app/singleplayer/page.js`** (`/local/page.js`를 기반으로 함): 사람=신원1, AI=신원2 고정(입장 바꿔 생각하기로 실제 담당 색이 바뀌어도 AI 로직은 `colorForPlayer` 기준으로 알아서 따라감 - `decideAiMove` 등에서 이미 처리). `useEffect`가 `isAiTurn(state,2)`일 때마다 `decideAiAction`으로 액션 하나를 계산해 살짝 지연(액션 종류별로 0.4~1.1초, "생각하는 척") 후 dispatch - 리듀서/기존 컴포넌트는 전혀 안 건드림. AI 패널(`canAct=false`, `peekedCard`/`pokerFaceReveal` 항상 숨김 - 온라인 모드에서 상대 정보를 숨기는 기존 관례와 동일하게 처리)과 AI의 증강 선택 화면(카드 내용 안 보여주고 "🤖 AI가 증강을 고르는 중..." placeholder만 - `RoomClient.jsx`의 "상대가 고르는 중" 패턴 재사용)은 사람이 못 건드리게 막음. 게임 종료 시 AI도 잠시 후 스스로 재도전 버튼을 누름. 홈페이지(`app/page.js`)에 "🤖 싱글플레이" 버튼 추가.

**검증**:
- Node로 AI vs AI 완전 자동 대국 스모크 테스트(스크래치패드 스크립트, 저장소에는 없음) - 300판 반복, 크래시/무한루프/멈춤 없이 전부 끝까지 진행되는 것 확인(평균 53틱/판, 승률 흑 55%/백 42%/기타 1% 정도로 큰 편향 없음). 이 과정에서 위 도장깨기 버그를 재현하고 수정함.
- Playwright로 실제 브라우저 라이브 검증: `/singleplayer` 접속 → 흑돌(사람) 시작 증강 선택 → AI(백돌) 시작 증강이 placeholder로 표시된 후 자동으로 선택되고 게임이 시작되는 것 → 사람이 클릭하면 AI가 자동으로 응수하는 것 → 4수마다 증강 선택 화면이 뜨고 카드를 고르면 정상 진행되는 것 → 실제로 게임이 끝까지 진행돼 승리 화면이 뜨는 것 → 사람이 재도전 버튼을 누르면 AI도 자동으로 눌러서 새 판(새 시작 증강 선택)이 시작되는 것까지 전부 확인.
- `next build` 통과(빌드 시 `/online` 프리렌더는 Supabase 환경변수가 없어서 실패하는데, 이는 이 세션 변경과 무관한 기존 환경 제약 - 더미 환경변수를 주면 전체 페이지 정상 빌드됨).
- 이번에도 `npm install`로 `node_modules`를 새로 설치해야 했고, `package-lock.json`이 의도치 않게 바뀐 걸 커밋 전에 `git checkout`으로 되돌림(세션 21과 동일한 상황).

**AI의 한계(의도적 트레이드오프, 다음에 참고)**:
- 미니맥스/몬테카를로 트리서치가 아니라 순수 휴리스틱(패턴 매칭 + 1수 앞 즉시 승리/차단)이라, 여러 수 앞을 내다보는 복잡한 함정(예: 이중 위협을 미리 깔아두는 수)은 못 읽음 - 캐주얼한 싱글플레이 상대로는 충분하지만 "최강 AI"는 아님.
- 증강 선택 점수(`AUGMENT_PICK_HINTS`)와 능력 우선순위(`ABILITY_PRIORITY`)는 68개 카드 각각을 정교하게 튜닝한 게 아니라 등급 기반 기본값 + 대략적인 손질이라, 특정 카드를 사람 고수만큼 예리하게 활용하지는 못함 - 밸런스 피드백을 받으면 두 표만 손보면 됨(로직 구조는 안 건드려도 됨).
- 향후 정말 새로운 **상호작용 형태**(기존 pendingTarget kind 어디에도 안 맞는 완전히 새로운 대상 지정 방식, 또는 카드 대상이 아닌 제3의 선택 UI)가 추가되면 dry-run 프레임워크가 자동으로 커버 못할 수 있음 - 그때는 `resolvePendingTarget`의 후보 생성 부분만 확장하면 됨(우선순위/증강 채점 로직은 그대로 재사용 가능).

### 24. 버그 수정: 링 위에서 싸우자 + 폭주(+체크무늬) 조합에서 소프트락
사용자 제보: "링 위에서 싸우자"가 최종 8x8까지 좁혀진 상태(+ 체크무늬로 짝수 칸만 가능한 상태)에서 폭주까지 겹쳐 있으면, 실제로 둘 수 있는 칸이 전부 채워져도 게임이 끝나지 않고 멈춰버림.

- **원인**: 물량전(무승부 방지) 판정이 `isBoardFull(newBoard)` — 보드 15x15 전체 225칸이 다 찼는지만 확인 — 을 썼는데, 링 바깥 영역(좁혀진 뒤로 다시는 못 둠)·체크무늬 홀수 칸(원천 착수 불가)은 "실제로 둘 수 있는 칸"이 전부 찬 뒤에도 영원히 빈 칸(0)으로 남아있어서 `isBoardFull`이 절대 `true`가 될 수 없었음. 그 결과 아무도 못 이기고 무승부/물량전 판정도 안 나서 게임이 영원히 멈춤 - 특히 폭주가 같이 켜져 있으면 클릭해도 `pickRandomLegalCell`이 둘 곳을 못 찾아 `return state`(조용한 무반응)만 반복돼서 완전한 소프트락이 됨. (역병으로 생긴 영구 죽은 칸(`deadCells`)도 같은 종류의 문제라 같이 처리함.) 링/체크무늬 중 하나만 켜져 있어도 같은 원인으로 발생하는 문제였음 - 폭주는 이걸 "조용한 멈춤"으로 증폭시킬 뿐, 근본 원인은 물량전/무승부 판정 쪽이었음.
- **수정**: `gomokuEngine.js`에 `isBoardEffectivelyFull(board, ringBounds, checkerboardActive, deadCells)` 추가 - 빈 칸이 있어도 그 칸이 링 바깥/체크무늬 홀수/죽은 칸이면 "이미 다 찬 것"으로 간주하고 건너뜀. `gameReducer.js`의 `CLICK_CELL`에서 기존 `isBoardFull(newBoard)` 호출을 이걸로 교체(링 경계는 이번 수로 갱신된 `stateAfterCapture.placementClock` 기준으로 재계산).
- **검증**: Node로 재현 스크립트 작성 - 수정 전 코드에 대고 돌려서 "실제로 둘 수 있는 마지막 한 칸"을 채워도 `gameOver`가 `false`로 남는 소프트락을 먼저 재현 확인 → 수정 후 같은 시나리오에서 `gameOver: true, winMessage: "무승부! (보드가 가득 찼어요)"`로 정상 종료되는 것 확인. `next build`도 통과(더미 Supabase 환경변수로 `/online`까지 전체 페이지 빌드 확인).

| discard | 파기 | 1회 사용 - 보유한 증강 카드 하나를 버리고 같은 등급의 새 카드로 교체 |

### 도박 전용 가짜 카드 (2, AUGMENTS 풀에는 없음)
| id | 이름 | 효과 |
|---|---|---|
| gambleMixed | 실버 2개 + 골드 1개 획득 | 무작위 실버 증강 2개 + 골드 증강 1개를 즉시 획득. 대신 다음 증강 선택 2번을 건너뜀 |
| gamblePrism1 | 프리즘 1개 획득 (45%) | 45% 확률로 무작위 프리즘 증강 1개 획득, 실패하면 아무것도 못 얻음. 어느 쪽이든 다음 증강 선택 2번을 건너뜀 |

## 알아두면 좋은 설계 패턴

- **회차 등급 공유**: `state.draftTierPlan[roundIndex]`에 그 회차 등급을 캐싱, 양쪽이 같은 roundIndex에 도달하면 같은 값을 재사용. `roundIndex = stonesPlaced[player]/4 - 1`
- **1회용 능력 공통 처리**: `oneTimeUsed[player][abilityId]` 플래그 + `ONE_TIME_ABILITY_IDS` 배열(거래가 한꺼번에 소모시킬 대상 목록)
- **면역 체크**: `checkImmunity(state, targetPlayer)` — 철옹성(무한)과 부적(1회성)을 하나로 통합 판정. 새 "공격형" 증강 만들 때 재사용
- **pendingTarget 패턴**: 칸 선택이 필요한 능력은 전부 `{player, kind, need, selected}` 구조. 2단계가 필요하면(재배치) `sourceCell` 같은 필드를 추가해서 같은 리듀서 케이스 안에서 분기
- **prison 필터링**: `getActiveAugmentIds(state, player)`를 거쳐야만 프리즘 무효화가 반영됨. 새 프리즘 증강을 만들 때 `ownedAugments[player].map(a=>a.id)` 대신 이걸 써야 교도소가 먹힘
- **물리적 신원 vs 논리적 색**: `RoomClient.jsx`의 `myRole`(localStorage에 저장된, 방을 나갔다 와도 고정인 신원)과 `myColor`(`toLogicalColor(myRole, gameState.colorFlipped)`로 계산하는, "지금 이 판에서 실제로 맡은 흑/백")를 분리해서 관리함. 재도전 때마다 색이 바뀌는 재도전 흑백 교대 기능 때문에 도입됨 - 게임 로직/권한 비교(현재 턴, 증강 선택 권한, 패널 표시 등)는 전부 `myColor` 기준, 신원 확인(claim, localStorage)만 `myRole` 기준
- **즉시발동 프리즘 패턴**: prison/battleRing/chaos/roleSwap처럼 "뽑는 순간 게임 끝까지 지속되는 전역 효과"는 `activateInstantAugments(state, grantedAugments, player)`에 한 줄만 추가하면 PICK_AUGMENT 본 흐름+거래+도박+동전 던지기+잠복+역감시 보너스+파기 교체 전부에서 자동으로 발동됨 (버그 8번 참고). 세션 21에서 timeCollapse(개인별 스냅샷)를 추가하며 `player` 인자를 새로 넣었으니, "누구 것인지"가 중요한 효과를 추가할 땐 이 인자를 반드시 실제 대상(잠복의 `victim`, 역감시의 `opponent` 등, 항상 "카드를 받는 사람"이지 "카드를 쓰게 한 사람"이 아닐 수 있음)으로 넘겨야 함
- **신원(identity) vs 돌 색(color) 분리**: `gameReducer.js`의 `player`/`currentPlayer`/`opponent`는 항상 "신원"(ownedAugments/stonesPlaced/lastMove 등을 조회하는 키) 이고, 실제로 board에 쓰거나 board에서 비교할 값은 반드시 `colorForPlayer(player, state.roleSwapActive)`를 거친 색이어야 함 (roleSwap 참고). 새 증강이 board를 직접 읽거나 쓸 때 이 변환을 빼먹으면 입장 바꿔 생각하기 발동 중에 조용히 깨짐
- **흑돌/백돌 전용 증강**: `AUGMENTS` 항목에 `colorOnly: 1|2` 필드를 추가하면 그 색을 지금 실제로 두고 있는 신원한테만 뽑기 풀에 등장함. 새 뽑기 경로를 추가할 때는 `drawFromPool`/`drawOneOfTier`/`drawSeveralOfTier`(또는 `barter`처럼 `AUGMENTS`를 직접 필터링하는 곳)에 반드시 `colorForPlayer(player, state.roleSwapActive)`를 `myColor` 인자로 넘겨야 함 - 빼먹으면 전용 카드가 엉뚱한 색 플레이어에게도 뽑히는 조용한 버그가 생김
- **퀘스트 증강**: `AUGMENTS` 항목에 `quest: true`를 붙이면 `AugmentPanel`에 🎯 배지가 붙음(로직에는 안 씀). 실제 발동 로직은 잠복/역감시와 같은 "조건 감지 → `triggerXxxQuestIfNeeded(state, player)` 헬퍼가 patch 반환 → 해당 지점에서 병합" 패턴을 그대로 재사용. 돌 제거처럼 여러 지점(포위 제거/습격/돌 제거/역병/붕괴)에서 똑같이 발동해야 하는 조건이면, 각 지점에서 실제로 제거가 "성사됐는지"(면역/여진에 막히지 않았는지) 확인한 뒤에만 헬퍼를 호출해야 함 - `survivor`(생존자) 참고
- **시작 증강(0수) 드래프트**: `initialGameState()`가 빈 상태를 만들자마자 `buildStartAugmentChoices(base, 1)`을 합쳐서 반환하는 방식으로, 첫 렌더부터 이미 흑돌의 `augmentSelect`가 켜져 있게 만듦(`CLICK_CELL`이 이미 `augmentSelect` 켜져 있으면 막는 기존 가드를 그대로 재사용). 흑→백 순서 체이닝은 `PICK_AUGMENT`의 마지막 공통 return에서 `state.augmentSelect.isStartDraft` 분기로 처리 - 새로운 카드가 픽 시점에 즉시 칸 지정/별도 화면으로 이어지는 타입(banZone/permaBlock/gamble류)이면 이 체이닝이 깨지니, 그런 신규 카드를 추가할 땐 `START_DRAFT_EXCLUDE_IDS`에도 추가할지 검토할 것
- **온라인 전용 증강**: `AUGMENTS` 항목에 `onlineOnly: true`를 붙이면 로컬 모드 뽑기 풀에서 항상 제외됨. `initialGameState(isOnlineMode)`가 `state.isOnlineMode`를 저장하고, `getModeExcludeIds(state)`(`state.isOnlineMode ? [] : ONLINE_ONLY_IDS`)를 모든 뽑기 경로(`buildAugmentChoices`/`buildStartAugmentChoices`/`REROLL_SLOT`/잠복·생존자·역감시·도박·동전던지기·먼저보기·벼랑끝)의 hardExclude에 섞어 넣는 방식 - `colorOnly`와 똑같은 자리에 나란히 적용하면 됨. 방을 새로 만드는 곳(`app/online/page.js`, `app/online/quick/page.js`)에서 `initialGameState(true)`를 호출해야 하고, 로컬(`app/local/page.js`)은 `initialGameState(false)`. `REQUEST_REMATCH`도 재시작 시 `initialGameState(state.isOnlineMode)`로 값을 이어받아야 재도전 후에도 모드가 안 풀림
- **카드 강화(감정/appraisal) 패턴**: 무작위로 새 카드를 주는 대신, 보유한 카드 "그 자체"를 강화하고 싶으면 `ENHANCEABLE_AUGMENT_IDS`에 id를 등록하고 `enhanceAugment(baseAugment)`로 `{...card, name: name+"+", enhanced:true}` 객체를 만들어 `ownedAugments[player]`의 해당 인덱스를 교체하면 됨(id는 그대로 유지되므로 `ownedIds.includes(id)` 같은 기존 소유 체크는 안 건드려도 계속 작동함). 실제 수치 반영은 각 효과 발동 지점에서 `state.ownedAugments[player].find(a=>a.id==="xxx")?.enhanced`로 개별 확인해야 함(예: banZone/watchtower의 turnsLeft, balance/lateBloomer/noYield/counterWatch/stockpile/brink의 임계값·확률·배수) - **착수 횟수에 영향을 주는 카드(양수겹침/질풍노도/도미노 등)는 절대 이 목록에 넣지 말 것** (이 세션의 핵심 밸런스 원칙을 무력화함)

## 알려진 제약/후속 검토 거리

- `relocate`/`collapse`/`prison` 등 구조적으로 새로운 메커니즘은 실제 대국에서 뽑힐 때까지 라이브 테스트가 제한적이었음 (역병만 end-to-end로 직접 검증함)
- **roleSwap 진짜 메커니즘 + threatRadar 선 잇기 리워크(세션 12)**: Node 리듀서 격리 테스트와 `next build`로만 검증됨 - 이 세션에선 다른 대화가 이미 이 프로젝트 디렉터리에서 dev 서버를 띄워둔 상태라 직접 브라우저 프리뷰를 못 띄웠음. 다음에 만질 일이 있으면 실제로 roleSwap을 뽑아서 (1) 상대 턴에 놓인 돌이 진짜 내 색이 아닌 색으로 표시되는지 (2) 증강 패널 타이틀이 실제 담당 색과 일치하는지 (3) 위험 감지가 빈 칸이 아니라 돌들을 잇는 선으로 뜨는지 브라우저에서 직접 확인해볼 것
- `역병`은 원안(승리 판정에서 그 돌만 영구 제외)이 아니라 "제거 + 칸 영구 봉쇄"로 단순화 구현됨 — 필요하면 원안대로 재구현 가능하지만 모든 win-check 함수를 고쳐야 함
- 온라인 모드는 각 클라이언트가 로컬로 리듀서를 돌리고 결과만 Supabase에 푸시하는 구조라, 이론상 네트워크 지연 시 레이스가 있을 수 있음 (턴제라 실제로는 거의 안 부딪힘, 지금까지 실제 버그 리포트는 없음)
- `.claude/launch.json`에 `npm run dev` 기준 미리보기 서버 설정이 있음 (포트 3000)
- **재도전 흑백 교대(colorFlipped)**: 리듀서 로직은 Node로 격리 테스트했지만, 실제 두 브라우저(Supabase 방 하나에 두 명 접속) 간 E2E는 계정 두 개가 필요해서 라이브 검증은 못 했음 - 코드 리뷰 수준의 확신만 있음. 다음에 온라인 모드를 만질 일이 있으면 실제로 재도전 두 번 눌러서 흑/백이 바뀌는지 확인해볼 것
- **착수 제한시간(30초) 타이머**: 로컬 모드는 라이브로 검증(3초로 임시 단축해서 자동 착수 확인)했지만, 온라인 모드는 "지금 차례인 클라이언트만 TIMEOUT을 쏜다"는 로직을 코드 리뷰로만 확인함 - 실제 네트워크 지연 상황에서 두 클라이언트가 동시에 쏘는 경쟁 상태가 없는지는 실제 두 브라우저로 확인 안 됨
- **시작 증강 드래프트(세션 18)**: 로컬 모드는 브라우저로 처음부터 끝까지(흑돌 선택→백돌 선택→첫 착수) 라이브 검증 완료. 온라인 모드는 "방 생성 직후 흑돌에게 시작 증강 화면이 뜨는지, 아직 안 들어온 백돌 자리가 정상적으로 대기하는지"를 실제 두 브라우저로는 확인 못했음(코드 리뷰 + Node 테스트만) - 다음에 온라인 모드를 만질 일이 있으면 확인해볼 것
- **정복자(conqueror) 퀘스트**: 실제 대국에서 중앙 3x3을 채우는 상황이 흔치 않아 Node 격리 테스트로만 검증(생존자는 5개 트리거 지점 전부 + 붕괴 동시발동까지 확인했지만, 정복자는 조건 자체가 하나뿐이라 상대적으로 단순함)
- **안개/체크무늬/예지/벼랑끝/감정/파기/노즈도르무(세션 19)**: Node 격리 테스트(23개) + `next build` + 로컬 모드 실제 브라우저 심층 플레이(임시 디버그 훅으로 8개 카드 전부 강제로 뽑아서 사용)까지 완료. 이 과정에서 안개의 `backdrop-filter`가 Tailwind v4/Lightning CSS 빌드에 의해 조용히 제거되는 실제 버그를 발견해서 인라인 style로 우회 수정함. 예지의 실제 캔버스 하이라이트 모양, 벼랑끝의 실제 격차-3 발동, 안개의 온라인 모드 두 브라우저 간 격리만 아직 미확인 - 다음에 온라인 모드를 만질 일이 있으면 안개부터 확인해볼 것
